# Studio Chat 整体体验升级

## 1. 文档目的

Studio Chat 的体验问题不只发生在“Agent 如何提问”，也发生在“Agent 如何展示执行过程和结果”。如果只优化其中一侧，用户仍然会遇到完整链路上的割裂：

- 开始前：系统重复提问，抽象风格难以判断。
- 运行中：文字、工具调用和文件操作连续堆叠，重点不清晰。
- 完成后：过程与结果权重接近，用户需要滚动查找产物和下一步。

因此，整体升级以一次任务的完整生命周期为对象，而不是只修改某个表单或消息组件。

## 2. 用户任务链路

Studio Chat 的核心链路分为五个阶段：

1. **进入**：用户从 Home、模板、历史项目或自由输入进入 Studio。
2. **确认**：Agent 读取已有上下文，只补问完成任务所必需的信息。
3. **执行**：Agent 规划并运行任务，展示必要进度、异常和待确认事项。
4. **交付**：Chat 突出最终结果、产物、采用的关键决策和下一步。
5. **迭代**：用户针对结果继续修改，而不是重新经历完整 discovery。

每个阶段都应只有一个明确的主动作。问题卡、运行进度、错误恢复和下一步推荐不能同时竞争用户注意力。

## 3. 核心问题

### 3.1 输入侧：意图确认成本高

- 已知信息没有充分继承，进入 Studio 后重复提问。
- 固定表单出现过早、问题过多，用户尚未看到价值就要先做决策。
- 风格主要依赖抽象文字，难以预测 Deck 或 Prototype 的最终效果。
- 问题、提交反馈和答案采用情况不在同一条对话时间线上。

### 3.2 输出侧：回复缺少任务层级

- Agent 的文字、思考、工具调用和文件变化按事件顺序平铺。
- 运行过程与最终结果视觉权重接近，长任务难以快速扫描。
- 任务完成后过程仍然全部展开，持续占据 Chat 高度。
- 简单任务也可能暴露过多执行细节，但错误、审批和风险又不能被隐藏。

### 3.3 共同问题：当前动作不够明确

- 问题未回答时可能同时出现下一步推荐。
- 运行中可能同时出现多个状态、操作入口和过程卡片。
- 完成后结果、产物、错误和后续建议缺少稳定优先级。
- 用户难以判断任务当前处于确认、执行、等待还是完成状态。

## 4. 总体目标

1. **以任务为中心**：Chat 展示用户正在完成的任务，而不是协议、工具和事件流水。
2. **渐进披露**：默认只展示当前需要的信息，细节在用户需要时展开。
3. **上下文连续**：Home 输入、问题、回答、执行和结果留在同一条时间线。
4. **主动作唯一**：每个任务状态只突出一个最重要的下一动作。
5. **结果优先**：任务完成后优先展示产物、结论和可继续操作。
6. **异常不隐藏**：错误、权限、审批和风险信息始终保持可见、可恢复。

## 5. 方案拆分

### 5.1 子方案 A：视觉化提问与意图确认

详细方案见 [`studio-chat-visual-discovery.md`](./studio-chat-visual-discovery.md)。

负责：

- 已知信息与缺失决策判断。
- 按入口决定是否提问。
- Deck / Prototype 专属视觉风格选择。
- Question Form 完整内联到 Chat。
- Submit / Skip、回答摘要和历史恢复。
- 提问漏斗与风格选择数据验证。

### 5.2 子方案 B：Assistant 回复信息架构

详细方案见 [`studio-chat-response-information-architecture.md`](./studio-chat-response-information-architecture.md)。

负责：

- 将执行事件聚合成用户可理解的任务组。
- 运行中展示当前任务与关键进度。
- 完成后收起过程，突出结果、产物和下一步。
- 保持错误、审批和待用户操作信息可见。
- 支持展开查看完整过程与文件修改。

### 5.3 共享基础

两个子方案共同依赖：

- 项目、入口、模板和 Design System 上下文。
- Assistant 消息、run、task、tool 和 artifact 的稳定关联 ID。
- Chat 内统一的状态优先级和消息层级。
- 曝光、操作、完成、失败和后续行为的漏斗数据。

## 6. 状态优先级

同一时刻按以下顺序决定 Chat 的主状态：

1. **等待用户回答或审批**：展示问题或审批操作，不展示普通下一步推荐。
2. **执行失败且可恢复**：展示错误原因和恢复动作。
3. **正在执行**：展示当前任务与关键进度，不提前展示完成态操作。
4. **执行完成**：展示结果、产物和下一步，过程默认收起。
5. **空闲可迭代**：保留简洁上下文，等待用户继续输入。

## 7. 交付边界

两个子方案共享总目标，但采用独立评审和独立开发 PR：

- 输入侧先验证“问题是否更少、更准、更直观”。
- 输出侧再验证“过程是否更易扫描、结果是否更容易找到”。
- 云端风格目录、动态提问策略和 Agent `styleSpec` 执行不阻塞前端原型。
- 回复聚合不改变底层 Agent 事件协议，第一阶段优先在现有事件上建立展示层。

## 8. 分阶段路线

### 第一阶段：输入侧闭环

- 问题全部内联 Chat。
- Deck / Prototype 静态视觉卡。
- Submit / Skip、已回答摘要。
- 问题回合不展示下一步推荐。
- 保留现有提问曝光、选择和提交埋点。

### 第二阶段：输出侧原型

- 对现有文字、工具调用和文件操作建立任务分组。
- 运行中展示当前任务，完成后收起过程。
- 错误、审批和用户操作保持展开。
- 增加任务组曝光、展开和结果操作埋点。

### 第三阶段：上下文与服务化

- Home 到 Studio 的完整上下文传递。
- 动态提问策略与问题去重。
- 云端视觉风格目录和版本管理。
- 输入决策、Agent 执行和最终结果的数据闭环。

## 9. 整体验收指标

- 首次可用产物耗时下降。
- 用户在确认阶段的有效提交率提升，平均问题数和提交耗时下降。
- 完成态消息默认高度和平均滚动距离下降。
- 用户定位产物、错误和下一步操作的耗时下降。
- 不降低 run 成功率、错误恢复率、审批完成率和产物打开率。
- 回答问题后，用户能在同一时间线确认答案、观察执行并找到结果。
