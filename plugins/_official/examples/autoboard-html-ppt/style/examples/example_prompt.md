# 示例提示词

> **适用规则版本**：v2.0 · 包含卡片高度控制、页面填充与留白、内容密度判断、同组卡片对齐协议、异构模块对齐、同一内容带对齐协议等全套约束。
> AI 在生成任何页面之前，必须先阅读并遵守本文件末尾的「**设计规则约束**」章节。

## R00 · 页面组件精准选择协议

> 本规则优先于所有页面示例和视觉规则。
> 必须先完成组件判断，再读取对应组件的详细生成规则。
> 禁止先看哪个示例视觉相似，就直接套用哪个模板。

---

### 一、判断顺序

每一页必须按照以下顺序处理：

1. 完整提取原始页面正式内容；
2. 判断页面角色；
3. 判断页面主要信息关系；
4. 统计模块、指标、图表和阶段数量；
5. 检查候选组件的必填字段；
6. 排除不满足条件的组件；
7. 检查组件内容容量；
8. 输出组件选择结果；
9. 最后才读取选中组件的详细生成规则并生成 HTML。

禁止同时读取全部组件的详细 HTML 后，凭视觉相似度选择。

---

### 二、页面角色

`pageRole` 只允许为：

- `cover`：封面
- `contents`：目录
- `section`：章节过渡
- `content`：正式内容页
- `closing`：封尾

页面角色不匹配时，组件直接排除。

例如：
- 正式业务内容页不得选择 `CoverPage`；
- 三个并列策略不得因为数量为 3 而选择 `ContentsPage`；
- 仍有业务数据和策略内容时不得选择 `ClosingPage`。

---

### 三、主要信息关系

每页必须识别一个 `primaryRelation`：

| 值 | 含义 |
|---|---|
| `independent-metrics` | 多个独立指标 |
| `parallel` | 同级并列 |
| `comparison` | 横向对比 |
| `process` | 前后流程 |
| `progression` | 阶段递进 |
| `cause-effect` | 因果关系 |
| `background-solution-result` | 背景—方法—结果 |
| `problem-strategy-result` | 问题—策略—结果 |
| `dual-chain` | 两条链路共同驱动结果 |
| `matrix` | 多个分类组成能力矩阵 |
| `summary` | 结束或总结 |

组件选择必须以 `primaryRelation` 为主，不能以单个关键词或卡片数量为主。

例如：
- 页面有三个数字，但主要表达"背景—方法—结果"，应选择案例页；
- 页面有三列，但三列存在时间先后，应选择阶段递进页；
- 页面有图表，但重点是三个独立业务成绩，应选择成绩单页。

---

### 四、组件准入条件

| 页面组件 | 必须具备 | 排除条件 | 推荐容量 |
|---|---|---|---|
| `CoverPage` | 开场主题、标题，可含副标题、时间、演讲人 | 存在多个正式业务模块、图表或策略 | 主标题 1 个，辅助信息 0～3 项 |
| `ContentsPage` | 2～4 个同级章节，包含章节名 | 内容是策略、指标、能力或流程，而非章节导航 | 2～4 章 |
| `SectionDividerPage` | 单一章节主题或章节编号 | 存在复杂正文和业务数据 | 1 个章节标题 |
| `MetricOverviewPage` | 2～4 个相互独立的大指标 | 存在完整流程、案例过程或每项独立复杂图表 | 2～4 个指标 |
| `CaseStudyPage` | 同一案例包含背景/问题、方法/策略、结果 | 三个模块只是同级并列，没有案例过程 | 3 个主模块，1～4 个结果指标 |
| `StrategyPanoramaPage` | 存在痛点、策略、收益三个区域及对应关系 | 只有同级策略，或没有痛点与结果 | 2～4 组痛点与策略 |
| `DualChainStrategyPage` | 明确存在两条链路或两个因素共同驱动结果 | 单一流程、普通两栏或三个同级主题 | 左右各 2～4 个节点 |
| `ThreeColumnStrategyPage` | 恰好三个同级主题，字段结构相近 | 存在时间顺序、因果、流程或阶段递进 | 3 个同级模块 |
| `SceneProgressionPage` | 存在明确时间、阶段、成熟度或场景递进 | 三个模块只是同级分类 | 2～4 个阶段 |
| `CapabilityMatrixPage` | 多个同级能力分类，每类包含若干能力项 | 存在前后流程或问题解法关系 | 3～5 个分类 |
| `GrowthReportPage` | 每个指标同时包含大数字、趋势/占比图和解释 | 只有简单数字，没有独立图表 | 2～3 个完整指标卡 |
| `ClosingPage` | 感谢、合作邀请、结束语、二维码或联系方式 | 仍有未表达完的业务内容 | 1 个结束主题，0～3 项辅助信息 |

---

### 五、必填字段检查

组件必须根据原稿中真实存在的内容判断是否可用。

**`CaseStudyPage`** 必须至少找到：
- `context`：背景或问题
- `method`：方法或策略
- `result`：结果或业务影响

其中任一项缺失时，不得强行选择 `CaseStudyPage`。

**`DualChainStrategyPage`** 必须至少找到：
- `leftChain`
- `rightChain`
- `sharedResult` 或明确的两链路关系

如果只找到左右两组并列内容，但没有双链路关系，不得选择。

**`GrowthReportPage`** 中，每个指标卡必须至少找到：
- `metricName`
- `metricValue`
- `chartData` 或原稿已有趋势信息
- `explanation`

缺少真实图表数据时，禁止自行编造趋势图。

---

### 六、禁止为了填模板补造内容

原始 PPT 没有提供的内容不得自动生成，包括：

- 虚构数据、虚构同比变化、虚构趋势图、虚构总结、虚构标签；
- 虚构案例背景、虚构问题、策略或结果；
- 为了填满三个卡片而拆分或复制内容；
- 为了凑满三个指标而增加新指标。

组件中的可选区域，在原稿没有对应内容时应隐藏，不得填充假内容。

**组件服务于原稿，原稿不得为了填满组件而被改写。**

---

### 七、组件容量检查

选择组件后必须检查：

- 原始模块数量是否超过组件容量；
- 每个模块的文字量是否超过对应卡片容量；
- 是否存在无法放入组件的正式内容；
- 所有数字、单位、注释和关系是否有对应位置。

容量不足时按以下顺序处理：

1. 使用同页面类型下容量更大的变体；
2. 调整卡片数量或列数；
3. 拆成连续页面；
4. 保留来源追踪关系。

禁止：删除原始内容 / 总结替代原文 / 缩小字号硬塞 / 选择语义错误但容量更大的组件。

---

### 八、组件选择置信度

每页输出 `componentConfidence`：

| 值 | 含义 |
|---|---|
| `high` | 组件完全符合页面关系和内容字段 |
| `medium` | 有两个相近候选，但其中一个更匹配 |
| `low` | 现有组件均不能完整承载内容 |

当置信度为 `low` 时：
- 不得强行套现有模板；
- 使用通用安全布局；
- 或输出两个候选组件及差异；
- 保留全部原始内容。

---

### 十、HTML 根节点必须声明组件元数据

每一个完整页面的 `.slide` 根节点必须添加以下 `data-*` 属性：

```html
<div
  class="slide"
  data-page-type="DualChainStrategyPage"
  data-layout-component="DualChainLayout"
  data-page-variant="CTRCVRDriven"
  data-primary-relation="dual-chain"
  data-required-fields="leftChain,rightChain,sharedResult"
  data-left-node-range="2-4"
  data-right-node-range="2-4"
  data-forbidden-relations="parallel,progression,matrix"
>
  <!-- 页面内容 -->
</div>
```

禁止省略这些属性；它们是后续内容审计、覆盖率检查和质量报告的依据。

---

### 十一、图标规范

图标统一使用 Lucide / Tabler 风格线性 SVG。

禁止使用 emoji、卡通贴纸和风格不统一的自绘图标。

## 示例 3：生成双链路策略拆解页

### 3-A 完整页面（整页调用）

请基于以下内容生成一页 3696×1008 超宽大屏 PPT，使用「超宽大屏商务增长风格」。

- 底部总结（3项）：过去半年拉回 CTR · 下一阶段提升 CVR · 推动 SCVR 持续提升

输出要求：

6. 配色：浅暖白背景（#FAF8F5）、青绿主色（#4DBFB0）用于圆形轮廓与高亮元素、香槟金辅色（#C9A96E）用于序号角标与装饰线、白色卡片带轻阴影（box-shadow: 0 4px 16px rgba(0,0,0,0.06)）。
7. 字号：主标题 72px，公式行 32px，定义行 26px，卡片标题 30px，要点正文 22px，圆内主标 52px，底部文字 28px。
8. 所有文本、图标说明、流程节点均为可编辑 DOM；圆形、卡片、箭头均用纯 CSS/SVG 实现，不使用外部图片。

# 系统定义

> 本章节定义本样式系统的适用范围、物理规格、风格关键词和超宽屏设计总原则。AI 在生成任何页面之前，必须先读完本章。

## 0.1 适用场景

本样式系统适用于：

- 会场主屏
- 发布会大屏
- 品牌营销大会
- 商业化业务汇报
- 品牌案例展示
- 数据策略发布
- 增长成果展示
- 超宽横幅型 PPT / KV 画面

---

## 0.2 固定画布规格

| 属性 | 规格 |
|------|------|
| 物理尺寸 | 11m × 3m |
| 像素分辨率 | **3696 × 1008 px** |
| 画面比例 | 11:3（约 3.67:1） |
| 画布方向 | 超宽横屏 |

> 所有 HTML 页面必须严格使用 `width:3696px; height:1008px`，不得使用百分比或 viewport 单位作为画布尺寸。

---

## 0.3 风格关键词

高级、明亮、克制、商务、轻科技、柔光感、品牌感、数据可信、生意增长、确定性、低信息密度、中心聚焦。

---

## 0.4 风格总原则

整体画面应传达一种**专业可信、增长明确、轻盈高级、适合会场大屏观看**的品牌商务视觉。

不追求炫技式未来科技感，不使用复杂的赛博视觉语言，而是通过：
- 浅暖色背景
- 青绿 + 香槟金双主色
- 柔和流线氛围
- 大标题中心聚焦
- 白色圆角卡片
- 大数字 / 大结论表达

建立"生意确定性"的视觉感受。

---

## 1.1 构图原则

由于本画布为超宽比例，必须按照**大屏主视觉**设计，而不是将普通 16:9 PPT 横向拉长。

核心原则：

| 原则 | 说明 |
|------|------|
| 中间集中表达 | 核心信息聚焦在画面中央 |
| 左右延展氛围 | 两侧放辅助图形与装饰性信息 |
| 模块横向展开 | 多模块并排横向排列，不纵向堆叠 |
| 留白优先 | 主动保留大面积留白，不填满每一个角落 |
| 远距离可读 | 标题 ≥ 72px，正文 ≥ 24px，关键数字 ≥ 96px |

---

## 1.2 视觉重心

建议将主视觉重心控制在**页面中央 50%–60%** 区域内。

- **中心区域**适合放：页面主标题、核心结论、核心案例标题、核心策略结构、关键数字结果、主卡片组。
- **左右两侧**适合放：氛围流线、辅助图形、次级模块、辅助标签、装饰性信息延展。

---

## 1.3 安全区

| 方向 | 边距 |
|------|------|
| 左安全边距 | 220 px |
| 右安全边距 | 220 px |
| 上安全边距 | 90 px |
| 下安全边距 | 90 px |

推荐核心内容区：

| 属性 | 数值 |
|------|------|
| 内容区起始 X | 220 px |
| 内容区结束 X | 3476 px |
| 可用内容区宽度 | **3256 px** |
| 内容区起始 Y | 90 px |
| 内容区结束 Y | 918 px |
| 可用内容区高度 | **828 px** |

> CSS 实现：`.safe-zone { position:absolute; left:220px; right:220px; top:90px; bottom:90px; }`
> 所有正式内容（标题、文字、卡片、图表、数字）必须完整落在安全区内，不得越界超过 2px。

**Logo 安全距离（§1.3 补充规则）：**

品牌 Logo（`.fixed-brand-logo`）位于 `left:96px; top:54px`，尺寸约 170×45px，底边约 **99px**。

安全区顶部为 90px，距 Logo 底边仅 9px，因此内容页需主动保留顶部间隙：

| 页面类型 | 顶部最小内边距 | 实现方式 |
|---------|--------------|---------|
| 多列布局页（三列、双列等）| `var(--space-40)`（40px）| 最外层容器 `padding-top: var(--space-40)` |
| 单列内容页（有页面标题）| `var(--space-32)`（32px）| 标题区 `padding-top: var(--space-32)` |
| 封面页 / 封尾页 | 不约束 | 以全屏背景图+居中文字为主 |

> ❌ 禁止：内容页第一行元素直接贴 `top:90px` 安全区起始位置，必须留出与 Logo 的视觉分离空间。

## 1.4 信息密度原则

超宽大屏必须比普通 PPT **更克制**：

| 密度控制项 | 规范 |
|-----------|------|
| 每页文字块数 | ≤ 6 块（含标题） |
| 每页主要卡片数 | ≤ 5 张（超过 5 张需拆分页面） |
| 每张卡片正文字数 | ≤ 80 字 |
| 每行字符数 | ≤ 28 字（正文 24px 时） |
| 最小字号 | 20px（注释/来源标注可降至 18px） |

> 信息密度优先级：**可读性 > 信息完整性 > 视觉美观**。宁可拆成两页，不压缩字号。

---

## 1.5 内容页主标题规则（硬约束）

**每一张内容页（`data-page-role="content"`）必须在安全区内包含一个可见的页面主标题。**

| 规则项 | 规范 |
|--------|------|
| 主标题字号 | 使用 `h2.title-xl`（72px）或行内等效 `font-size:72px; font-weight:700` |
| 主标题位置 | 安全区内顶部，外层容器 `padding-top:var(--space-40)` 后第一个元素 |
| 主标题垂直对齐 | 标题行容器使用 `align-items:center`，使主标题与副说明**垂直居中对齐** |
| 主标题结构 | `<h2 class="title-xl">页面主题</h2>`，可附一个 `<span>` 副说明（24–28px，`color:var(--text-muted)`） |
| 内容区 | 主标题之下用 `flex:1; min-height:0` 的容器承载主体内容 |

**❌ 禁止以下替代方式：**
- 用卡片 banner 条替代主标题（如三列页各列顶部的色块横幅）
- 用 28px 以下的标注文字替代主标题
- 完全省略页面主标题，让内容直接填满安全区
- 标题行使用 `align-items:baseline`（会导致不同高度文字上下错位）

**布局模板（内容页标准骨架）：**

```html
<div class="safe-zone">
  <!-- 外层 flex-col -->
  <div class="flex-col" style="height:100%; padding-top:var(--space-40); gap:var(--space-16);">

    <!-- ① 页面主标题行（必须存在，align-items:center 垂直居中）-->
    <div style="flex-shrink:0; display:flex; align-items:center; gap:var(--space-16);">
      <h2 class="title-xl" style="margin:0;">页面主题</h2>
      <span style="font-size:26px; color:var(--text-muted);">副说明（可选）</span>
    </div>

    <!-- ② 内容主区（flex:1 填满剩余高度） -->
    <div style="flex:1; min-height:0; /* 根据布局选择 flex-row / grid-col 等 */">
      <!-- 卡片、列表、图表等主体内容 -->
    </div>

  </div>
</div>
```

> 封面页（`data-page-role="cover"`）、目录页（`"contents"`）、封尾页（`"closing"`）不受此约束，可使用 `h1.title-cover`（148px）等超大标题。

## 1.6 Page Title Layout Rules（Title Group 布局规范）

页面主标题采用 **「Title Group」布局**，而不是单独排版文字。

**Title Group 包含：**
- ① **主标题（Primary Title）**：`h2.title-xl`，72px，font-weight:700
- ② **副标题（Subtitle，可选）**：`<span>`，字号约为主标题的 30–40%（推荐 22–28px），`color:var(--text-muted)`

---

### 垂直居中（Optical Center 对齐）

| 规则 | 说明 |
|------|------|
| 主标题是视觉中心 | 副标题围绕主标题进行垂直居中对齐 |
| 副标题中心线 = 主标题文字高度中心 | 使用 `align-items:center` 实现，主标题加 `line-height:1` 消除隐式行高 |
| 禁止副标题贴底 | ❌ 不允许 `align-self:flex-end` 或 `align-items:baseline` |
| 禁止按 Baseline 对齐 | ❌ `align-items:baseline` 会因字号差异导致副标题上移或下沉 |
| 采用视觉中心对齐 | `align-items:center` + 主标题 `line-height:1` |

### 排列方式

| 属性 | 规范 |
|------|------|
| 主标题 | 在左 |
| 副标题 | 在右 |
| 间距 | `gap: 12–20px`（推荐 `var(--space-16)`） |
| 副标题字号 | 约为主标题的 30–40%（72px 主标题 → 22–28px 副标题） |

### 整体对齐

- 整个 Title Group 作为**一个整体单元**，不拆分排版
- Title Group 与页面内容区**左边缘对齐**
- Title Group **高度自适应**（由 `padding: var(--space-24) 0` 撑开），不允许副标题单独漂移

### 禁止项

| ❌ 禁止 | 原因 |
|---------|------|
| 副标题贴近标题底部 | 破坏视觉重心，视觉不稳定 |
| 副标题顶部对齐（`align-items:flex-start`）| 大小字号高度差导致副标题悬空 |
| 按字体 Baseline 对齐（`align-items:baseline`）| Baseline 偏下，副标题视觉下沉 |
| 副标题高度超过主标题中线 | 视觉层级颠倒 |

### 标准代码模板

```html
<!-- ✅ 正确：Title Group，align-items:center + 主标题 line-height:1 -->
<div style="flex-shrink:0; display:flex; align-items:center;
            gap:var(--space-16); padding:var(--space-24) 0;">
  <h2 class="title-xl" style="margin:0; line-height:1;">页面主题</h2>
  <span style="font-size:26px; color:var(--text-muted); font-weight:400;">副说明（可选）</span>
</div>

<!-- ❌ 错误：baseline 对齐 -->
<div style="display:flex; align-items:baseline; ...">...</div>

<!-- ❌ 错误：副标题单独 align-self:flex-end -->
<span style="align-self:flex-end;">...</span>
```

> **关键技巧**：主标题 `line-height:1` 可消除 `line-height:1.15` 默认值带来的约 10.8px 隐式上下空白，使 72px 字体在容器中的视觉中心与数学中心一致。


## 3.4 标题规则

**主标题要求：**
- 居中排版
- 单页不超过 **2 行**
- 字间距略舒展（`letter-spacing: -0.01em ~ 0.02em`）
- 行高适中（`line-height: 1.1~1.25`）
- 文字尽量结论化（直接呈现结果/判断，不写过程）

**副标题要求：**
- 位于主标题下方，`margin-top: 12–20px`
- 信息简洁（≤1行，≤24字）
- 颜色弱于主标题（使用 `var(--text-muted)` 或 `var(--text-secondary)`）
- 字号为主标题的 30–40%（如主标题 72px，副标题 24–28px）
- 不超过 1 行

## 5.6 分栏适用场景

| 分栏比例 | 适用场景 |

| **8:4** | 主内容 + 侧栏说明；图片主导型页面 |
| **7:5** | 叙述文本 + 辅助图表；信息不对称对比 |
| **4:4:4** | 三项等权能力 / 步骤 / 指标 |
| **3:6:3** | 两侧辅助 + 中心主区；对称枢纽布局 |

**分栏规则：**

1. 避免机械 6:6 贯穿全 deck；若所有页面都 6:6 等分，会丧失节奏感和主次关系。
2. 同一 deck 中不同页面应使用不同分栏比例，形成视觉节奏。
3. 图片可以跨越栏线一次（编辑突破），制造有意的构图张力；其他文字 / 数字 / 卡片元素仍须对齐栏线。
4. 分栏边界用 Grid 轨道变量实现，禁止逐元素用 `margin` 或 `left` 手动拼凑对齐。

**CSS 实现：**

```css
:root {
  --col: calc((100% - 11 * var(--col-gap)) / 12);
  --col-gap: 24px;
}

/* 常用分栏 */
.grid-8-4   { grid-template-columns: calc(8 * var(--col) + 7 * var(--col-gap))
                                     calc(4 * var(--col) + 3 * var(--col-gap)); }
.grid-7-5   { grid-template-columns: calc(7 * var(--col) + 6 * var(--col-gap))
                                     calc(5 * var(--col) + 4 * var(--col-gap)); }
.grid-4-4-4 { grid-template-columns: repeat(3, calc(4 * var(--col) + 3 * var(--col-gap))); }
.grid-3-6-3 { grid-template-columns: calc(3 * var(--col) + 2 * var(--col-gap))
                                     calc(6 * var(--col) + 5 * var(--col-gap))
                                     calc(3 * var(--col) + 2 * var(--col-gap)); }
```

---

## 5.7 基线节奏与 margin 使用约束

1. **正文基线节奏统一**：相邻文本块使用相同的 `line-height` 和间距变量（`--space-*`），不得用随意 margin 微调到"差不多齐"。
2. **同级文本块间距一致**：`margin-top / margin-bottom` 必须来自同一变量或同一数值，不得出现一处 18px、另一处 19px 的手动补偿。
3. **跨模块对齐靠 Grid**：跨模块的基线对齐通过共同父级 Grid 行轨道实现，而非逐元素 margin 偏移。
4. **间距台阶约束**：只允许在以下台阶中选取，不得使用非台阶数值（如 17px、21px、35px）：

```
--space-8   --space-12  --space-16  --space-24
--space-32  --space-40  --space-48
```

## 6.2 卡片高度控制（硬约束）

| 约束项 | 规范 |
|--------|------|
| 默认高度 | `height: auto`；禁止默认使用 `height: 100%`、`flex: 1` 或过大的 `min-height` |
| 内容较少时 | 禁止将卡片强制拉伸到画布底部 |
| 正文 < 120 字 | 卡片高度不得超过画布高度的 55%（554px） |

**卡片组对齐规则：**

- 纵向组：左、右边缘对齐，外框等宽。
- 矩阵组：各行等高、各列等宽，行列间距一致。
- 内容差异只能通过内部弹性空间、固定底部结论区或精简文案处理；外框对齐优先于内部内容对齐。
- 主次卡、瀑布流、时间轴或刻意非对称构图可例外，但必须在页面计划中说明设计意图。

## 8.3 形状语言与大版式几何动作（硬约束）

**允许的基础形状：**

| 手法 | 说明 | 实现参考 | 尺寸约束 |
|------|------|---------|---------|
| 通高色列 | 纵向等宽色块从顶贯穿至底，锚定节奏 | `width:8px–40px; height:100%; background:var(--gold-main)` | 宽度 ≤ 画布宽度 3%（约 110px） |
| 巨型数字 | 核心数字放大为背景装饰层 | `font-size:160px–200px; opacity:.08–.18` | opacity 0.06–0.20，不遮文字 |
| 粗横杠 | 横线横穿版面，分隔或强调 | `height:4px–12px; background:linear-gradient(90deg,…)` | 高度 ≤ 画布高度 3%（约 30px） |
| 单条斜线 | 贯穿版面的斜线（30°–60°），引导视线 | SVG `<line>` 或 `transform:rotate()` | `stroke-width:1px–3px` |

**使用约束：**

1. 每页最多使用 **一个** 大尺度几何动作，禁止叠加。
2. 必须位于内容层之下（`z-index` 低于内容），不得遮挡文字、数字、卡片主体或连接线。
3. 所有大尺度几何元素必须使用品牌色板颜色（金色 `#D5AE79`、青绿 `#4F9B90`、深色系），不得引入新颜色。

# 设计规则约束

> AI 在生成任何页面之前，必须先完整阅读以下全部规则，并在输出前逐条自检。

---

## R1 · 卡片高度控制

**禁止在内容较少时将卡片强制拉伸至画布底部。**

1. 禁止默认使用 `height: 100%`、`flex: 1` 或过大的 `min-height` 作为卡片首选高度。
2. 卡片应优先使用 `height: auto`。
3. 当单张卡片正文少于 120 字时，卡片高度不得超过画布高度的 55%。
4. 卡片有效内容占卡片高度的比例应不低于 55%。
5. 页面剩余留白应分布在模块**外部**，而不是集中在卡片**内部**。
6. 同组卡片需要等高时，以内容最多的卡片为基准，不得以画布剩余高度为基准。

---

## R2 · 页面填充与留白控制

**页面不得仅依赖卡片高度填充画布。**

当文字内容较少时：
- 禁止通过增大卡片 `height`、`min-height` 或 `padding` 强行撑满页面。
- 禁止添加无意义长文、重复信息或纯装饰图形填充空间。
- 必须优先重构信息层级，增加图表、流程、关系、数据对比或主题视觉。

**页面应满足以下视觉占比：**

| 区域 | 占画布高度 |
|------|-----------|
| 标题区 | 10%–18% |
| 主体内容区 | 45%–65% |
| 辅助视觉区 | 15%–30% |
| 连续无功能留白（上限） | ≤ 25% |
| 有效信息 + 有效视觉覆盖面积 | 55%–75% |

当卡片自适应后导致页面内容不足时，依次采取：
1. 放大核心信息，而不是放大容器；
2. 将关键数据升级为主视觉；
3. 将并列模块改为流程、路径或关系结构；
4. 增加与主题相关的信息图形；
5. 调整页面垂直重心；
6. 最后才调整模块尺寸。

同一页面中，留白必须**分散**在标题、模块和视觉元素之间，不得集中形成大面积空洞区域。

---

## R3 · 内容密度判断

AI 在生成页面前，应先判断内容密度：

| 密度 | 对应布局策略 |
|------|-------------|
| 高密度内容 | 紧凑卡片、分栏、表格 |
| 中密度内容 | 卡片 + 数据图形组合 |
| 低密度内容 | 少量大模块、主视觉数据、流程图或主题图形 |

**禁止所有内容密度都套用三等分卡片布局。**

---

## R4 · 同组卡片对齐协议

当两个或以上卡片在视觉上属于同一组、同一行、同一列或同一矩阵时，必须先建立统一的对齐关系，再进行内部排版。

### 一、先识别分组

- 同一行卡片属于**横向卡片组**。
- 同一列卡片属于**纵向卡片组**。
- 多行多列卡片属于**矩阵卡片组**。
- 只有真正属于同一语义层级的卡片才统一对齐。
- 不同层级、不同功能或刻意强调的主卡片，可不强制等尺寸，但仍需遵守共同基线。

### 二、横向卡片组规则

同一行卡片必须满足：
- 顶边对齐；底边对齐；外框高度一致；
- 垂直方向不得单独偏移；
- 卡片之间的水平间距一致；
- 卡片内部内容差异不得改变外框对齐关系。

```
top(card_i)    = top(card_j)
bottom(card_i) = bottom(card_j)
height(card_i) = height(card_j)
```

### 三、纵向卡片组规则

同一列卡片必须满足：
- 左边缘对齐；右边缘对齐；外框宽度一致；
- 水平方向不得单独偏移；
- 卡片之间的垂直间距一致。

```
left(card_i)  = left(card_j)
right(card_i) = right(card_j)
width(card_i) = width(card_j)
```

### 四、矩阵卡片组规则

- 同一行共享相同顶部和底部基线；
- 同一列共享相同左侧和右侧基线；
- 行高由该行统一决定；列宽由该列统一决定；
- 行间距一致；列间距一致；
- **禁止每张卡片独立计算位置。**

### 五、父级统一控制

- 卡片位置和尺寸必须优先由共同父容器控制。
- 优先使用 Grid 或 Flex，不允许无必要地逐张绝对定位。
- 同组卡片禁止分别设置独立的 `top`、`left`、`margin-top`、`margin-left`、`translateX`、`translateY`。
- 同组卡片不得混用不同的 `box-sizing`、边框宽度、外边距或尺寸计算方式。
- 同组卡片应共享统一的尺寸变量、间距变量和对齐方式。

### 六、外框对齐优先于内容对齐

先保证卡片外框边界统一，再处理内部标签、标题、正文、图表和数字的排布。内容多少不同，只允许影响卡片内部的留白和排列，不得破坏卡片外框对齐。

### 七、内部锚点对齐

当同组卡片结构相似时，还应建立内部锚点：
- 标签顶部对齐；
- 标题起始线对齐；
- 正文起始线对齐；
- 底部按钮或结论区对齐。

可为缺失内容保留结构占位，但不得使用无意义文本填充。

### 八、允许例外

只有以下情况允许不等高或不等宽：
- 明确的主次卡设计；
- 瀑布流布局；
- 时间轴错落布局；
- 刻意的不对称构图；
- 大卡带小卡的组合结构。

使用例外时，必须明确说明设计意图，不能因内容长度不同而自然失控。

### 九、生成后自动检查

- 同组卡片边界误差是否小于 1–2px；
- 是否存在单卡片纵向或横向偏移；
- 是否因内容长度导致外框尺寸不一致；
- 是否存在视觉上接近但实际未对齐的情况；
- 是否使用统一间距和统一尺寸变量。

### 附：内容不等高时的处理顺序

1. 统一外框尺寸；
2. 内容顶部对齐；
3. 通过内部弹性空间吸收高度差；
4. 底部操作区或结论区固定到底部；
5. 必要时缩短冗余文字；
6. **禁止通过移动整个卡片解决内容高度差。**

---

## R5 · 异构模块对齐规则

当页面同时包含列表、卡片、箭头、数据指标、图表等不同类型模块时，不得简单要求所有元素等宽或等高。

**必须先建立以下对齐锚点：**

### 1. 外边界锚点
- 同一逻辑层级模块共享顶部或左侧基线。
- 总结型模块至少与主体模块共享一个外边界。

### 2. 行轨道锚点
- 存在逐项对应关系的两个列表，必须共享相同的行数、行高和行间距。
- 对应项的顶部、底部和中心线必须一致。

### 3. 中心线锚点
- 箭头、连接线、转折符号必须对准其连接对象的中心，而不是页面中心。
- 核心数字、图表或结论内容块，应与主体信息区域的视觉中心对齐。

### 4. 语义关系优先
- 有对应关系的元素，优先**逐行对齐**。
- 有因果关系的元素，优先通过**连接线和中心线对齐**。
- 有总结关系的元素，优先通过**共同顶部和内容中心对齐**。

### 5. 不同类型模块
- 不同模块可以不同宽、不同高。
- 但必须至少共享一个边界锚点和一个内容锚点。
- 不允许完全独立摆放。

### 6. 禁止补偿式定位
- 禁止通过单独设置 `margin`、`top`、`left`、`translate` 修复对齐。
- 应回到父级 Grid、Flex、行轨道和对齐变量中解决。

### 7. 生成后检查
- 对应项是否逐行齐平；
- 箭头是否真正对准连接对象；
- 总结模块是否与主体模块建立共同基线；
- 是否存在看似接近、实际相差数像素的伪对齐；
- 不同模块之间是否形成明确的视觉阅读路径。

## R6 · 同一内容带中的异构模块与卡片对齐规则

当页面中多个一级模块位于同一横向内容带内时，无论模块内部是多张列表卡片、单张数据卡、图表卡、总结卡或其他异构结构，都必须先建立统一的外部对齐关系，再处理各自内部排版。

### 一、内容带统一高度 H

首先定义统一内容带高度 H，并由共同父级 Grid 或 Flex 容器统一控制。所有一级模块必须满足：

```
top(module_i)    = top(module_j)
bottom(module_i) = bottom(module_j)
height(module_i) = height(module_j) = H
```

**H 的限制：**
- 不得由某一张异常偏高的卡片决定；
- 不得使用页面剩余高度、`100vh`、独立 `min-height`、`flex: 1` 或单模块固定高度自动撑满；
- H 应根据页面内容密度统一确定，通常占画布高度的 **35%–55%**。

### 二、标题条 + 多卡片模块的内部计算

```
cardsAreaHeight = H - titleHeight - titleToCardsGap

singleCardHeight = (cardsAreaHeight - cardGap × (cardCount - 1)) / cardCount
```

- 标题条使用统一固定高度；
- 卡片区域内部使用 Grid，卡片等分剩余高度；
- 卡片之间间距一致，不得继续按各自文字内容自动收缩。

### 三、逐项对应关系的两组卡片

```
top(leftCard_n)    = top(rightCard_n)
bottom(leftCard_n) = bottom(rightCard_n)
height(leftCard_n) = height(rightCard_n)
```

### 四、单体成果卡 / 图表卡

- 外框高度同样必须等于 H；
- 视觉强调通过放大核心数字、图表、标题层级或颜色对比实现，**不得通过扩大空容器制造重点**；
- 模块内部内容块可以垂直居中，但外框必须与同一内容带中的其他一级模块共享顶部和底部边界。

### 五、对齐锚点要求

不同类型模块不要求内部结构完全一致，但至少必须共享两个对齐锚点：
1. **外部边界锚点**：共同顶部、共同底部、共同整体高度；
2. **内部内容锚点**：列表使用共同轨道，数据卡使用内容中心，箭头和连接线使用关联对象的中心线。

连接箭头、流程符号或过渡文字**不参与一级模块高度计算**。它们必须对准所连接模块的内容中心或对应行中心，不得按照整页中心随意定位。

### 六、禁止的修正方式

禁止通过以下方式逐个修正对齐：
- 单独设置 `margin-top` 或 `margin-bottom`；
- 单独设置 `top`、`bottom`、`left`、`right`；
- 使用 `translateX` 或 `translateY` 微调；
- 给某个模块设置独立 `height` 或 `min-height`；
- 使用绝对定位模拟视觉对齐；
- 通过增加无意义 `padding` 撑高模块。

### 七、出现对齐问题时的处理顺序

1. 修正共同父级内容带高度 H；
2. 修正父级 Grid/Flex 的 `align-items: stretch`；
3. 确保所有一级模块使用 `height: 100%` 和 `box-sizing: border-box`；
4. 修正复合模块内部的 Grid 行轨道；
5. 修正标题条高度、卡片高度和间距变量；
6. 最后处理模块内部文字和图形位置。

**不得通过移动单个元素掩盖父级结构问题。**

### 八、推荐实现结构

```css
.content-band {
  display: grid;
  align-items: stretch;
  height: var(--content-band-height); /* 例如 55vh */
}

.level-one-module {
  height: 100%;
  min-height: 0;
  box-sizing: border-box;
}

.list-module {
  display: grid;
  grid-template-rows:
    var(--group-title-height)
    minmax(0, 1fr);
  row-gap: var(--group-title-gap);
}

.card-list {
  display: grid;
  grid-template-rows:
    repeat(var(--card-count), minmax(0, 1fr));
  row-gap: var(--card-gap);
}

.item-card,
.result-card,
.chart-card {
  height: 100%;
  min-height: 0;
  box-sizing: border-box;
}
```

### 九、生成后检查（使用 DOM 几何数据，不得只凭肉眼）

使用 `getBoundingClientRect()` 检查所有一级模块和对应卡片。

**允许误差：**

| 检查项 | 允许误差 |
|-------|---------|
| 一级模块 top 差值 | ≤ 2px |
| 一级模块 bottom 差值 | ≤ 2px |
| 一级模块 height 差值 | ≤ 2px |
| 同组卡片高度差 | ≤ 2px |
| 对应行卡片顶部和底部差值 | ≤ 2px |

若任一项超过误差，必须继续修改父级 Grid、内容带高度或内部行轨道，**不得使用局部位移进行补偿**。

**最终目标：** 同一内容带中的多个模块形成完整闭合的矩形关系，顶部在同一水平线，底部在同一水平线，整体高度一致；多卡片模块通过等分内部空间填满内容带，单体成果模块通过内部内容居中填满内容带。

## R8 · 溢出检查与修复协议

AI 在生成或修复页面时，如遇到内容溢出，**必须先判断溢出原因，再按对应方案处理**。禁止不加判断地缩字号、压边距或裁切内容。

---

### 一、先诊断溢出类型

**类型 A：元素越界** — 内容量本身不多，但元素超出了安全区边界（贴边、越界、被裁切）。

**类型 B：内容过量** — 内容量客观超过单页容纳上限，无论怎么排版都放不下。

---

### 二、类型 A 处理：强制约束安全区

当溢出原因是元素越过安全区时，**强制所有核心内容约束在安全区内**：

| 方向 | 最小边距 |
|------|---------|
| 左侧 | ≥ 220px |
| 右侧 | ≥ 220px |
| 顶部 | ≥ 90px |
| 底部 | ≥ 90px |

具体约束：
- 标题、正文、数字、标签**不允许贴边或越界**；
- 使用 `.safe-zone`（`left:220px; right:220px; top:90px; height:828px`）或等效约束包裹所有核心内容；
- 装饰性背景图、流线、光晕**不受安全区限制**，可全出血；
- Logo 组件位于安全区外侧（`left:96px; top:54px`），属于独立层，不受此约束。

---

### 三、类型 B 处理：重构内容，不压缩

当溢出原因是**内容量过多**时，**严禁**通过以下方式硬塞内容：
- 缩小安全区边距；
- 压缩字号至规定最小值以下；
- 截断、省略或改写原文内容；
- 增大卡片 padding 以外的无意义压缩。

**必须按以下顺序处理：**

1. **重新分组内容** — 将长列表、多段落拆为逻辑组，减少单屏信息密度；
2. **更换更合适的布局** — 从三等分卡片改为双列、流程、表格或时间轴等更高密度布局；
3. **必要时拆成两页或多页** — 保持原文内容完整，每页聚焦一个逻辑段落，加页码延续标记；
4. **原文内容不得省略、截断或改写** — 数字、单位、专有名词、因果关系必须完整保留。

---

### 四、修复后的验收标准

| 验收项 | 要求 |
|-------|------|
| 100% 画布下 | 无溢出、无遮挡、无裁切 |
| 缩放预览下 | 仍然清晰可读，文字不重叠 |
| 安全区内 | 所有核心内容均在边界内 |
| 内容完整性 | 原文无删减、无改写 |
| 字号下限 | 正文 ≥ 18px，标注 ≥ 16px |

---

### 五、禁止的修复行为

| 禁止项 | 原因 |
|-------|------|
| 缩小全局字号 | 破坏可读性，远距离不可辨 |
| 压缩安全区边距 < 220px / 90px | 违反视觉安全规范 |
| 截断或省略正文内容 | 信息完整性不可妥协 |
| 修改数字、单位、专有名词 | 可能造成信息失真 |
| 用 overflow: hidden 遮盖溢出 | 掩盖问题而非解决问题 |
| 用绝对定位强行压入画布 | 可能导致层叠遮挡 |

## R9 · 内容拆解与信息权重协议

AI 在开始布局前，必须先完成内容拆解，标注每条信息的类型、权重、表现形式和关系。

### 一、拆解维度

| 维度 | 含义 | 可选值 |
|------|------|--------|
| type | 内容类型 | title / data / description / stage / comparison / case / chart / conclusion |
| weight | 信息权重 | core（必须突出）/ normal（常规展示）/ aux（辅助补充） |
| format | 表现形式 | text / number / percentage / image / chart / icon |
| relation | 内容关系 | parallel / compare / cause / aggregate / sequence |

### 二、权重判断规则

- 核心数字（增长率、关键占比、突破性成果）→ **core**
- 核心结论（整页最想传达的判断）→ **core**
- 模块标题、场景说明 → **normal**
- 注释、来源、辅助标签 → **aux**
- **core 项不超过 3 个**，超过时降级为 normal

### 三、信息层级分配

| 层级 | 角色 | 视觉权重 |
|------|------|---------|
| L0 | 页面主标题（唯一） | 最大 |
| L1 | 页面核心结论（1–2 个） | 仅次于 L0 |
| L2 | 一级模块标题 | 区分功能区 |
| L3 | 核心数据/核心概念 | 数字比说明大 3–5 倍 |
| L4 | 正文说明/图表说明 | 常规文字 |
| L5 | 注释/来源/辅助标签 | 最小最淡 |

### 四、层级驱动空间分配

- 含 L3 数据的模块 → 宽度倾向放大（如 1.5:1 非等分）
- 含 L4 长说明的模块 → 高度倾向增加
- 纯 L5 辅助的模块 → 宽度可压缩为标签条

## R10 · 布局模板选择协议

根据内容的主要关系，选择匹配的布局模板。

### 布局模板总览（T01–T10）

| 模板 | 名称 | 适用场景 | main 列分配 |
|------|------|---------|------------|
| T01 | 目录/章节导航页 | 报告目录、会议议程 | 等分，章节数决定列数 |
| T02 | 核心数据展示页 | 多项业绩、增长指标 | 等分卡片，≥7 个拆页 |
| T03 | 中心概念/双边连接页 | 平台连接、生态协同 | 25:50:25 |
| T04 | 左右策略对比页 | 过去与现在、优化前后 | 50:50，≥4 项改表格 |
| T05 | 场景→模型→调整→结果页 | 算法模型、投放优化 | 17:42:25:17 |
| T06 | 阶段演进页 | 冷启到成熟、能力升级 | 等分阶段，≥6 个拆页 |
| T07 | 曲线路径/战略路线图 | 长期战略、技术演进 | 路径区 100%，3–6 节点 |
| T08 | 能力矩阵页 | AI 能力总览、产品能力地图 | 3–5 列等宽 |
| T09 | 案例展示页 | 产品界面、广告样式 | 图片区 55%–75% |
| T10 | 未来展望/业务拓展页 | 下阶段规划、新业务 | 左主题 17%–25% + 右卡片 75%–83% |

### 不等分是常态

- 含 L3 数据的模块 → 宽度更大
- 图片/案例区域 → 占 55%–75%
- 对比页 → 等分
- 结论精炼 + 多卡片 → 1:4

### 混合关系处理

| 混合类型 | 处理方式 |
|----------|---------|
| 并列 + 因果 | 主模板用并列（T02/T08），因果用箭头或路径线串联 |
| 对比 + 数据 | 主模板用对比（T04），数据作为两侧结果卡 |
| 递进 + 聚合 | 主模板用递进（T06），中心节点作聚合锚点 |
| 数据 + 展望 | 主模板用数据（T02），展望结论放 footer |

## R11 · 容器承载系统

AI 在选择布局后，必须判断每类信息该落入哪种容器。容器是信息结构的一部分，不是装饰。

### 一、容器等级

| 等级 | 用途 | 信息权重 | 数量限制 |
|------|------|---------|---------|
| A 主容器 | 页面核心概念/主视觉 | core | 最多 1–2 个 |
| B 次容器 | 说明模块/辅助结果 | normal | 可重复，不抢过 A |
| C 辅助容器 | 标签/状态/编号 | aux | 只做索引补充 |

### 二、容器类型决策

```
内容是页面唯一核心概念    → 圆形 / 同心圆（A）
内容是一组完整模块        → 大圆角卡片（A/B）
内容是阶段/背景说明       → 轻量说明卡（B）
内容是标签/编号/状态      → 胶囊（C）
内容是一句话总结          → 长条承载（A/B）
页面需要强张力/方向感     → 异形描边承载（A/B）
辅助引导/留白构图         → 无底框结构（B/C）
```

### 三、容器与层级映射

| 层级 | 推荐容器 |
|------|---------|
| L0 | 无底框标题区 |
| L1 | 标题下方副结论 / 长条承载 |
| L2 | 卡片标题 / 栏目标题 |
| L3 | 圆形主容器 / 数据卡主数字 |
| L4 | 说明卡 / 卡片正文 |
| L5 | 胶囊标签 / 角落注释 |

## R12 · 组件组合模式库

以下 10 种高频组合模式，AI 可直接匹配调用。

### 模式 0：双主卡 + 中轴说明（dual_cards_with_center_description）

**适用**：两个对称角色，中间有桥梁性概念（广告主/媒体、输入/输出）

**列比例**：左卡 42% + 中轴 16% + 右卡 42%（column_ratio: "1:0.35:1"）

**中轴组件规则**：
- 不加大底框，居中对齐
- 宽度明显小于左右卡
- 内容：主概念 → 次概念 → 多行说明 → 连接符

---

### 模式 1：中心构图（圆形中心 + 双侧卡）

**适用**：平台连接、供需关系、生态协同

**列比例**：25:50:25（左角色卡 : 中央圆形 : 右角色卡）

---

### 模式 2：双圆公式（A × B 公式页）

**适用**：变量乘积、双能力拼接

**结构**：左圆 + 中间连接符 + 右圆，左右两侧或下方补能力卡

---

### 模式 3：大卡内嵌小组件

**适用**：模型能力、机制说明、复杂模块

**规则**：内部小组件（截图/图表/胶囊标签）依附大卡，不单独跑到页面层

---

### 模式 4：左引导 + 右卡片群

**适用**：未来展望、方向拓展

**列比例**：左引导 17%–25%（无底框）+ 右侧 3–5 张业务卡 75%–83%

---

### 模式 5：中央图片 + 双侧数据

**适用**：案例展示、UI 截图、产品素材

**列比例**：左数据 17%–20% + 中央图片组 60%–66% + 右数据 17%–20%

**规则**：图片自己就是主体，不强加底框

---

### 模式 6：路径 + 节点 + 旁注

**适用**：战略路线图、增长曲线、能力演进

**规则**：路径线作为主骨架，节点承载核心词，解释文字放节点外侧

---

### 模式 7：矩阵列 + 底部胶囊结论

**适用**：能力矩阵、分类总览

**结构**：3–5 列等宽栏 → 栏内：标题 → 条目 → 底部胶囊总结

**规则**：栏目间靠对齐和分隔线，不靠大卡片

---

### 模式 8：对称结果 + 异形描边框

**适用**：结果页、章节重点结论（信息量少但需要强张力）

**结构**：左结果框（镜像）+ 中央核心结论 + 右结果框（镜像）

---

### 模式 9：同心圆 + 侧数据卡

**适用**：用户价值分层、能力沉淀

**结构**：左侧主题引导 + 中央同心圆模型 + 右侧数据卡列

---

### 模式 9.5：三段流程 + 连接节点（triple_process_with_connector_nodes）

**适用**：三阶段流程、左→中→右能力推进

**列比例**：A模块 + 连接节点(0.18) + B模块(1.2) + 连接节点(0.18) + C模块（column_ratio: "1:0.18:1.2:0.18:1"）

**连接节点规则**：
- 箭头是独立组件，尺寸为普通 icon 的 1.5–2 倍
- 圆底完整托住箭头
- 位于两个主模块之间的中点，沿主模块视觉中心线对齐
- 两个连接节点样式和尺寸必须统一

---

### 模式 10：三卡数据 + 底部总结

**适用**：成绩页、增长页、三项结果总览

**规则**：三卡应同结构、同宽度、同层级；底部统一总结条

---

## R13 · 图片布局协议

当页面包含图片时，必须先判断图片角色，再选择组合布局。

### 一、图片角色判定

| 图片角色 | 判定特征 | 空间策略 |
|---------|---------|---------|
| **主体** | 产品界面、截图、数据面板 | 占 main 55%–75%，文字围绕 |
| **证据** | 图片旁有数据结果且数据是核心 | 图片跟随数据，不独立占大区域 |
| **说明** | 在流程节点或模型描述内部 | 嵌入说明模块，和文字并列 |
| **氛围** | 背景/品牌/装饰 | 不参与布局计算，叠加底层 |

### 二、图文组合布局

| 布局 | 比例 | 适用 |
|------|------|------|
| 图左 + 数据右（证据型） | 40 : 60 | 案例 + 增长数据 |
| 数据左 + 图右（镜像） | 55 : 45 | 增长指标 + 截图证据 |
| 图中央 + 数据双侧（主体型） | 17:60:17 | 多张界面 + 两侧关键数据 |
| 图上方 + 文字下方（说明型） | 55% 高 + 45% 高（列内子布局）| 模型能力页 |
| 图嵌入卡片（说明型） | 占卡片 40%–60% 高 | SDK 能力、产品功能 |
| 纯图片序列（主体型） | 图片横向序列 55%–75% 高 | 素材总览 |

### 三、多图排列规则

- **横向序列**：等宽、等间距（1%–2%），超过 6 张时分组
- **左右对比**：两张同比例（48%:48%），各配标题，底部统一结论
- **2×2 网格**：不超过 3×3，统一尺寸，每张配短标签

## R14 · 通用组件使用规范（C01–C10）

AI 在排布组件时，必须遵守以下组件结构规范。

### C01 页面标题

```
品牌占位 → 主标题 → 副标题/辅助口号 → 标题分隔线
```
- 每页一个主标题；标题过长时改写，不无限缩小

### C02 核心数据

```
核心数值 → 趋势符号 → 指标名称 → 时间范围 → 解释说明
```
- 一个组件只突出一个核心指标；数据与单位不可分离；多个数据组件应统一结构

### C03 标签

类型：章节 / 业务 / 时间 / 阶段 / 策略 / 编号 / 状态
- 标签不应替代标题，也不应承载长段落

### C04 内容卡片

```
卡片标题 → 卡片副标题 → 主体说明 → 图片/图表/图形 → 卡片结论
```
- 一个卡片只表达一个主要主题；标题→内容→结论保持固定顺序；同组卡片使用统一结构

### C05 图表

| 数据关系 | 推荐图表 |
|---------|---------|
| 趋势 | 折线图 |
| 前后差异 | 柱状图 |
| 比例 | 环形图/饼图 |
| 阶段 | 阶梯图/路径图 |
| 层级 | 同心圆/金字塔 |
| 关系 | 辐射图/网络图 |
| 规模 | 面积图/柱状图 |

- 图表必须服务于结论；只保留必要坐标轴；关键数据必须有视觉区分

### C06 图片/案例

```
图片 → 图片编号 → 图片分类 → 图片说明 → 案例结果
```
- 统一图片比例和间距；按时间/类型/流程排序；图片不应替代标题和结论

### C07 流程节点

```
节点编号 → 图标（可选）→ 节点名称 → 节点动作 → 节点说明 → 节点结果
```
状态：起点 / 进行中 / 当前 / 已完成 / 下一阶段 / 目标状态

### C08 路径线

- 路径线必须有方向；不能穿过核心文字；复杂关系改用矩阵或流程图

### C09 底部结论条

```
图标或箭头 → 一句话结论 → 关键数据 → 下一步方向
```
- 每页最多一个主要结论条；结论表达完整判断，不仅堆叠关键词；高度自适应

### C10 分隔组件

类型：水平 / 垂直 / 虚线 / 点线 / 模块留白 / 色块边界 / 中心轴线
- 分隔线用于表达结构；有方向关系时使用箭头；不能切断主要阅读路径

## R15 · 字号层级规范

AI 生成所有页面时，必须遵循以下字号层级，不得随意缩小或任意设定字号。

| 角色 | 推荐字号 |
|------|---------|
| 封面主标题 | 148px |
| 目录页主标题 | 82px |
| 常规页面主标题 | 72px |
| 一级模块标题 | 44px–28px |
| 二级模块标题 | 36px |
| 核心指标数字 | 48px、64px、72px、100px、200px |
| 正文 | 24px、28px、30px、32px、36px |
| 指标说明 | 26px、30px、48px |
| 注释说明 | 22px、28px |

**规则：**
1. 同一页面内，相同语义角色必须使用相同字号；
2. 核心指标数字根据重要程度和版式空间选择对应挡位，不得混用；
3. 正文字号至少为 24px，注释说明不得低于 22px；
4. 标题层级严格递降，不得出现二级标题大于一级标题的情况；
5. 溢出时优先缩短文字、拆分模块或改变布局，不得降低字号到规定值以下。

## R16 · 超宽画布卡片宽度、文本行长与图片裁切规则

超宽画布（11:3、32:9 等）只代表横向空间更大，**不代表单个内容模块应无限拉宽**。生成前必须先建立"内容最大宽度"约束。

### 一、文本卡片宽度

| 卡片类型 | 宽度上限 |
|---------|---------|
| 普通说明型文本卡片 | 画布宽度的 22%–32% |
| 信息较多的文本卡片 | 不超过画布宽度的 38% |
| 单栏长文本区域 | 不超过主体内容区的 45% |

```css
.text-card {
  width: 100%;
  max-width: var(--text-card-max-width);
}
```

禁止只使用无上限的 `1fr` 拉伸文本卡片。

超宽画布中优先使用：多列布局 / 左文右图 / 主卡+辅卡 / 短文本+数据视觉 / 多个紧凑内容单元。

### 二、正文行长规则

| 场景 | 单行上限 |
|------|---------|
| 中文正文（常规） | 约 22–36 个汉字 |
| 大屏演示场景 | 不超过约 40 个汉字 |

```css
.card-copy {
  max-width: 36em;   /* 或 max-width: 34ch; */
  line-height: 1.5;
}
```

- 说明型小字不得在超宽卡片中形成一整条横向长句；
- 文字过少时，应缩窄文本区，不得让文字贴左而右侧留大面积空白；
- 文字过多时，拆分层级/分栏/提炼内容，不得持续扩大卡片宽度；
- 标题、正文、标签可拥有不同最大宽度，正文不必跟随卡片整体宽度铺满。

### 三、图片卡片宽度与比例规则

每个图片容器必须具有：明确宽度、明确高度或 `aspect-ratio`、`overflow: hidden`、统一圆角、统一裁切方式。

```css
.image-card {
  width: 100%;
  height: 100%;
  overflow: hidden;
  border-radius: var(--card-radius);
}
.image-card img {
  width: 100%;
  height: 100%;
  object-fit: cover;
  object-position: center;
  display: block;
}
```

**严禁：**
- `object-fit: contain` 导致图片悬浮在容器中；
- 图片上下或左右出现无意义空白；
- 通过拉伸图片宽高填满容器。

### 四、图片必须优先使用 cover 裁切

当图片作为卡片主视觉、背景图、案例图、场景图或氛围图时，默认使用 `object-fit: cover`。

根据图片主体位置设置 `object-position`：

| 主体位置 | object-position |
|---------|----------------|
| 居中 | `center center` |
| 人物偏左 | `35% center` |
| 人物偏右 | `65% center` |
| 重点在上 | `center 30%` |
| 重点在下 | `center 70%` |

**禁止对所有图片机械使用完全相同的中心裁切**，AI 必须先判断主体位置，再确定裁切焦点。

### 五、图片比例选择规则

图片显示比例必须服从页面结构，不服从图片原始比例。

| 用途 | 推荐比例 |
|------|---------|
| 横向场景图 | 16:9、3:2 或 2:1 |
| 品牌氛围图 | 4:3 或 3:2 |
| 人物或产品竖图 | 3:4 或 4:5 |
| 超宽背景图 | 2.4:1–3.5:1 |
| 小型缩略图 | 1:1 |

同组图片必须共享相同显示比例，禁止出现一横一竖、高度不同、裁切方式不同或无法对齐的情况。

### 六、图文组合卡片规则

| 主导类型 | 图片占比 | 文本占比 |
|---------|---------|---------|
| 图片主导型 | 55%–70% | 30%–45% |
| 信息主导型 | 35%–45% | 55%–65% |
| 均衡型 | ~50% | ~50% |

超宽画布中不建议单侧超过模块宽度 75%。

```css
.media-card {
  display: grid;
  grid-template-columns:
    minmax(0, var(--image-max-width))
    minmax(0, var(--text-max-width));
  gap: var(--content-gap);
}
```

图片区和文本区都必须有独立最大宽度约束，不能仅依靠 `1fr 1fr` 无限扩展。

### 七、低内容量处理规则

当卡片内容较少时，**禁止**通过以下方式填满超宽空间：增加卡片宽度 / 拉长正文行长 / 增加无意义内边距 / 将小图片等比缩放后放在大容器中。

应按以下顺序处理：
1. 缩小单个卡片最大宽度；
2. 调整为紧凑组合；
3. 增加列间距或组间距；
4. 将关键图片放大并使用 cover 裁切；
5. 将数据、关键词或结论升级为视觉模块；
6. 使用不对称构图分配剩余空间；
7. 必要时增加与内容相关的图表或视觉表达。

### 八、卡片组最大宽度规则

```css
.card-group {
  width: min(100%, var(--group-max-width));
  margin-inline: auto;
  display: grid;
  gap: var(--card-gap);
}
```

超宽画布中，内容组可居中、偏左或偏右，但必须有明确最大宽度。**禁止主体内容从左安全区一直连续铺到右安全区。**

### 九、生成后检查

| 检查项 | 要求 |
|-------|------|
| 文本卡片行长 | 未形成超长行，未超过 40 字 |
| 图片填充 | 完整覆盖容器，无上下/左右空白 |
| 图片变形 | 无拉伸变形 |
| 图片主体 | 未被错误裁掉 |
| 同组图片 | 显示比例和高度一致 |
| 模块宽度 | 未超过预设最大宽度 |
| 画布填充方式 | 通过版式关系消化横向空间，非拉宽卡片 |

## R17 · 大会型 PPT 连接枢纽组件规则

本规则适用于大会演讲、行业峰会、品牌发布、招商推介、年度总结、战略发布、产品介绍和案例分享等场景。连接枢纽组件用于表达多角色、模块、能力、阶段、资源、数据或结论之间的关系。

### 一、适用场景

| 类型 | 示例 |
|------|------|
| 多方协同 | 品牌、平台、渠道、用户、合作伙伴 |
| 业务关系 | 供给、平台、需求 |
| 流程链路 | 输入、处理、输出 |
| 战略路径 | 现状、策略、结果 |
| 能力体系 | 核心能力连接多个子能力 |
| 营销链路 | 曝光、点击、转化、复购 |
| 增长模型 | 流量、效率、转化、收入 |
| 案例总结 | 挑战、方法、成果 |
| 组合关系 | 能力 A × 能力 B、产品 + 服务 |
| 阶段演进 | 过去、现在、未来 |

### 二、组件基础结构

一套完整的连接枢纽组件可包含（非全部必选）：
1. 核心节点
2. 外部业务模块
3. 连接线或关系符号
4. 连接锚点
5. 节奏节点
6. 辅助说明
7. 必要的环形、同心圆或路径装饰

**禁止为装饰而增加无意义圆环、节点和线条。**

### 三、核心节点规则

- 必须保持严格正圆，使用 `aspect-ratio: 1 / 1`；
- 不得因文字长度变成椭圆；
- 文字建议控制在 2–8 个汉字或简短缩写；
- 只放核心概念，不放长段说明；
- 节点直径必须设置最大值，不得因超宽画布无限放大；
- 视觉强调来自字号、字重、描边、背景层次或轻微阴影，**禁止**厚重发光、复杂立体效果或过度装饰。

### 四、外部模块规则

同一层级的外部模块必须共享统一的：高度/尺寸规则、圆角规则、描边规则、内边距、标题层级、连接锚点、与中心节点的间距。

外部模块中的文字应保持精简，每个模块优先表达一个核心概念。

### 五、连接关系类型

| 连接样式 | 语义 |
|---------|------|
| 实线 | 明确、稳定、直接关系 |
| 虚线 | 辅助、潜在、弱关联 |
| 点线 | 资源触点、传播路径、协同节点 |
| 单向箭头 | 流程、递进、输入输出 |
| 双向箭头 | 互动、反馈、双向协同 |
| 无箭头线 | 对等、共同组成 |
| 环形路径 | 循环、闭环、持续运营 |
| 运算符 × + = → | 组合、因果、转化关系 |
| 汇聚 | 多个节点指向一个中心 |
| 分发 | 一个中心连接多个外部节点 |

**同一页面中，相同语义必须使用相同连接样式。**

### 六、连接锚点规则

连接线必须从节点或模块的明确边界锚点出发，不得从文字区域、圆心或卡片内部随意出发。

当中心节点连接左右卡片时，三者垂直中心线必须对齐：
```
centerY(leftModule) = centerY(centerNode) = centerY(rightModule)
```

当中心节点连接上下模块时：
```
centerX(topModule) = centerX(centerNode) = centerX(bottomModule)
```

### 七、连接线几何规则

- 从真实边界出发，到真实边界结束；
- 不穿过主要文字和核心数字；
- 不无序交叉；
- 优先水平线、垂直线和规则折线，只有版式明确需要时才用斜线或曲线；
- 连接路径由 Grid、Flex 或 SVG 坐标系统控制，**禁止逐条用 `margin`、`top`、`left`、`translate` 肉眼调整**。

### 八、节奏节点规则

- 可在连接线上设置小圆点、空心点、短线或阶段标记，表达流转过程、阶段递进或多触点协同；
- 节奏节点应均匀分布，视觉权重弱于核心节点和业务模块；
- 一条普通连接线建议使用 0–3 个节奏节点，禁止过多造成装饰化和视觉噪声。

### 九、推荐结构变体（10 种）

| 变体 | 结构描述 | 适用场景 |
|------|---------|---------|
| **中心枢纽型** | 一个中心节点连接周围多个业务模块 | 多方协同、平台生态、资源整合 |
| **中继转换型** | 输入模块 → 中心节点 → 输出模块 | 数据处理、策略转化、流量承接 |
| **双节点协同型** | 节点 A × 节点 B | 两项能力组合、流量与效率 |
| **多节点流程型** | A → B → C → 结果 | 业务流程、用户旅程、增长路径 |
| **汇聚型** | 多个外部节点共同指向一个中心结果 | 多能力形成成果、多渠道汇聚 |
| **分发型** | 一个中心节点向多个外部模块分发 | 平台赋能、能力输出、服务覆盖 |
| **环形闭环型** | 多个节点沿圆环排列，形成循环 | 用户运营闭环、品牌增长闭环 |
| **分层同心圆型** | 中心概念在内层，外围表示能力/资源/生态层 | 能力体系、产品架构、用户圈层 |
| **主干分支型** | 一条主路径连接多个分支节点 | 战略主线、产品路线图、多业务分支 |
| **对比连接型** | 两侧展示不同对象，中间用枢纽或运算符连接 | 过去与未来、挑战与方案 |

**禁止所有页面重复同一种"左右卡片 + 中间圆形"结构。**

### 十、大会 PPT 视觉要求

- 核心概念需在 3–5 秒内被识别；
- 主要关系路径必须清晰；
- 连接线不得过细（建议线宽 ≥ 1.5px）；
- 辅助说明字号对应 R15 中注释说明（22px–28px），不得更小；

**单页建议数量：**
- 核心节点：1–3 个
- 普通外部节点：2–8 个
- 主要连接关系：1–6 条

超过此数量时，应拆页、分组或分阶段呈现。

### 十一、超宽画布规则

- 中心节点不得无限放大；
- 外部模块不得被推到画布两端；
- 节点组必须设置最大宽度（参照 R16）；
- **禁止用超长连接线填满画布**；
- **禁止让关联模块彼此距离过远**。

超宽画布优先使用：多组并列关系 / 左右分区 / 主关系+辅助关系 / 中心枢纽+两侧业务带 / 多阶段横向流程。

### 十二、多组件复用规则

同一套 PPT 中多次使用连接枢纽组件时，以下元素必须统一：

**必须统一：** 核心节点形状、节点边框、节点背景、连接线线宽、箭头样式、节奏节点样式、字体层级（遵循 R15）、圆角与阴影、连接锚点逻辑、主色与辅助色。

**允许变化：** 节点数量、连接方向、主次大小、关系符号、局部强调色、外部卡片内容、横/纵/环形/放射布局。

### 十三、主次关系规则

- 核心节点尺寸可比普通节点大 10%–25%；
- 核心节点可提高描边或背景对比；
- 普通节点不得缩小到影响阅读；
- 所有节点仍需共享明确中心线或结构轨道；
- 不得依靠随意放大制造主次。

### 十四、内容密度适配

| 密度 | 处理方式 |
|------|---------|
| 低密度 | 少量大节点，放大核心概念，关系线简洁，可加适量环形或同心圆 |
| 中密度 | 中心节点 + 3–6 个外部模块，明确主路径和辅助路径，一页一个核心关系 |
| 高密度 | 拆分为多个分组，使用分层或阶段呈现，必要时分两页或多页 |

### 十五、禁止事项

❌ 将圆形拉伸成椭圆  
❌ 节点文字过多  
❌ 连接线从文字中穿过  
❌ 连接线停在节点外形成断点  
❌ 多条线无序交叉  
❌ 所有关系都使用同一种线条  
❌ 节点与外部模块没有明确对齐关系  
❌ 超宽画布中使用超长连接线  
❌ 通过放大中心节点填满页面  
❌ 使用过重阴影、发光、厚描边  
❌ 每页都重复同一种中心圆结构  
❌ 逐个使用 `top`、`left`、`margin`、`translate` 修正位置  
❌ 为纯装饰增加无语义圆点、线条和圆环  

### 十六、生成后检查

| 检查项 | 要求 |
|-------|------|
| 核心关系 | 可快速理解 |
| 中心节点 | 为正圆或统一几何形 |
| 同类节点 | 尺寸一致 |
| 连接线 | 准确接触节点边界，不穿过文字 |
| 运算符位置 | 处于相邻节点中心之间 |
| 主路径 | 比辅助路径更清晰 |
| 节点组 | 未过度松散 |
| 超宽画布 | 无意义超长连接 |
| 组件风格 | 同套 PPT 中保持一致 |
| 结构适配 | 当前结构真正适合内容，非机械套用圆形 |

## R18 · 形状、卡片与装饰规范

### 允许的形状元素

- 圆角矩形、正圆；
- 直角小方点、点阵、规则网格；
- **一个大尺度几何动作**：通高色列、巨型数字、粗横杠、单条斜线；
- 标准几何图形构成的装饰区（如色块分区、色带背景）。

### 禁止的装饰行为

❌ 禁止为填空白加入无关图库照片  
❌ 禁止每页重复相同的"左图片右三卡"结构  
❌ 禁止为纯装饰增加无语义圆环、线条、圆点  
❌ 禁止使用过重阴影、发光效果、厚描边  
❌ 禁止使用复杂立体效果或拟物化装饰

## R19 · 网格系统与排版节奏规范

### 一、12 栏网格

页面使用 **12 栏网格**。常用分栏比例：

| 分栏 | 适用场景 |
|------|---------|
| 8:4 | 主内容 + 辅助说明 |
| 7:5 | 内容主导 + 数据佐证 |
| 4:4:4 | 三等分卡片/步骤 |
| 3:6:3 | 侧说明 + 中心主体 + 侧说明 |

**避免机械 6:6 贯穿整套 PPT。**

### 二、图片栏线规则

- 图片可以**跨越栏线一次**，形成有意的编辑突破；
- 其他元素仍需对齐栏线；
- 不得让多个元素同时跨越栏线造成结构混乱。

### 三、文本基线节奏

- 正文基线节奏统一；
- 相邻文本块**不得用随意 margin 微调到"差不多齐"**；
- 同组文字必须共享相同基线或顶线对齐。

### 四、横向主图比例

| 用途 | 推荐比例 |
|------|---------|
| 横向主图 | 16:10、4:3 或适合页面栏宽的自然比例 |
| 人物肖像 | 3:4 |
| 多图组 | 保持同高或共享基线 |

比例是构图建议，**不要求裁掉主体来强行匹配**。

### 五、生成图约束

生成图（AI 生图）**严禁**：
- 自带 PPT 标题、页眉、页脚、页码；
- 自带 caption 说明文字；
- 自带装饰边框；

这些编辑元素由页面 HTML 单独排版，生成图只提供纯净视觉内容。

## R20 · 安全区、Grid 与留白优先协议

本规则是所有页面布局的基础约束层，优先级高于局部样式决策。

---

### 一、安全区（硬边界）

画布尺寸：**3696 × 1008 px**

| 方向 | 固定值 | 百分比等效 |
|------|--------|-----------|
| Left | 220px | 3.25% |
| Right | 220px | 3.25% |
| Top | 90px | 4.76% (≈ 90/1008 × 100) |
| Bottom | 90px | 4.76% |

**可用内容区：3256 × 828 px**

规则：
- 所有正式内容（标题、正文、数字、卡片、图表、标签）**不得越过安全区超过 2px**，否则视为失败；
- 背景图、装饰光晕、色块氛围图不受此限制，可全出血；
- Logo 独立层位于安全区外侧（`left:96px; top:54px`），不受此约束；
- 若画布尺寸调整，按百分比重新计算，不得硬编码 px 值。

```css
.safe-zone {
  position: absolute;
  left: 3.25%;   /* = 220px @ 3696 */
  right: 3.25%;
  top: 4.76%;    /* = 90px @ 1008 */
  bottom: 4.76%;
  /* 或等效固定值版本 */
  /* left: 220px; right: 220px; top: 90px; bottom: 90px; */
}
```

---

### 二、12 栏 Grid 系统

页面内部不使用自由布局，一律基于 **12 栏网格**排列组件。

```css
.layout-grid {
  display: grid;
  grid-template-columns: repeat(12, 1fr);
  gap: 0.7%;   /* gutter = 0.7% of content area width */
}
```

**组件宽度由栏数决定，不得硬编码 px 宽度：**

| 分栏组合 | 适用场景 |
|---------|---------|
| 8 : 4 | 主内容 + 辅助说明 |
| 7 : 5 | 内容主导 + 数据佐证 |
| 4 : 4 : 4 | 三等分卡片 / 步骤 |
| 3 : 6 : 3 | 侧说明 + 中心主体 + 侧说明 |
| 6 : 6 | 仅对比页使用，避免机械贯穿全套 PPT |

禁止：
- `left: 425px; width: 760px` 这类写死 px 的布局；
- 绕开 Grid 直接用 `position: absolute` 定位内容模块；
- 用 `margin-left` / `translate` 微调列位置来对齐。

---

### 三、留白优先（Whitespace First）

超宽画布不是要被填满，而是让重要内容有呼吸空间。

**正确的留白分布：**
```
标题区（含内边距）
  ↓ 48–64px 间距
主体内容区
  ↓ 适当留白
图形/数据区
  ↓ 底部边距
```

**留白必须均匀分布**在标题、模块和视觉元素之间，不得集中形成大空洞。

---

### 四、布局规则速查表

| 分类 | 规则 |
|------|------|
| **页面安全区** | 左右 220px（3.25%）、上下 90px（4.76%），正式内容不允许越界超过 2px |
| **Grid** | 全局 12 栏 Grid，gutter 0.7%，组件按列宽布局，禁止硬编码 px 宽度 |
| **页面边距** | 统一使用安全区，不允许贴边 |
| **留白** | 留白优先，均匀分布，不集中形成大空洞 |
| **标题** | 默认居中，≤ 2 行，与主体保持 48–64px 间距 |
| **卡片间距** | 同级统一 24–40px |
| **卡片 Padding** | 普通卡 24–32px，大卡 32–40px |
| **间距 Token** | 仅允许使用 8 / 12 / 16 / 24 / 32 / 40 / 48px，禁止随意微调 |
| **卡片对齐** | 同组等高、等宽、统一 Gap；外框对齐优先于内容对齐 |
| **布局控制** | 一律由父级 Grid/Flex 控制，禁止单独 margin、translate、绝对定位补偿 |
| **溢出处理** | 优先换布局、分页或调整结构；禁止压缩安全区、负 margin 或减小内边距硬塞内容 |

---

### 五、禁止通过以下方式撑满页面

❌ 增加卡片高度  
❌ 增加 `padding` 至异常大  
❌ 增加 `min-height` 强行拉高  
❌ 增加假文字或空白段落  
❌ `margin` 随意微调到"差不多齐"  
❌ 通过 `position: absolute` + `top/left` 硬定位内容模块  

**页面不得仅依赖卡片高度填充画布。剩余空间应通过留白、视觉层级和构图关系处理。**

## R21 · 排版规则分类速查表

> 本表为所有页面排版的强制执行清单，AI 在生成每一页时须逐条核对。

| 分类 | 规则详述 |
|------|---------|
| **页面安全区** | 左右各 220px（3.25%），上下各 90px（4.76%）。所有正式内容（标题、卡片、数字、文字）不允许越界；背景图/装饰可出血。 |
| **Grid** | 全局采用 12 栏 Grid（gutter 0.7%），组件宽度由所占栏数决定，禁止硬编码 `width: XXpx` 定位内容列。 |
| **页面边距** | 统一以安全区作为页面边距基准，不允许内容贴边；禁止在安全区外放置正式内容。 |
| **页面留白** | 留白优先——超宽画布的剩余空间应均匀分布于各模块之间，不得集中形成大空洞；禁止用假内容/padding 填充空白。 |
| **标题** | 默认水平居中，行数 ≤ 2 行；标题区域与主体内容之间保持 **48–64px** 间距，不得紧贴。 |
| **卡片间距** | 同组/同级卡片的 gap 统一，取值范围 **24–40px**；不同组件之间的间距须保持一致，不随意微调。 |

| **Margin** | 仅允许使用统一间距 Token：**8 / 12 / 16 / 24 / 32 / 40 / 48 px**；禁止出现非 Token 值（如 13px、27px、53px）。 |
| **卡片对齐** | 同组卡片：等高、等宽、统一 Gap；以外框对齐为优先，不允许靠内容长短决定卡片尺寸。 |
| **布局方式** | 一律由父级 Grid 或 Flex 控制子元素位置；禁止依赖单独的 `margin`、`transform: translate`、`position: absolute` 来补偿对齐偏差。 |
| **溢出处理** | 内容超量时优先更换布局（缩减列数/分页/重组结构）；禁止通过压缩安全区、负 margin、减小字号或内边距来硬塞内容。 |

## R23 · 固定品牌信息禁用规则

## R25 · 双向关系图（左-中-右三区）布局优化规则

适用于：SDK竞争力升级、CTR×CVR驱动SCVR提升等"双向关系图"页面（左区输入 → 中心枢纽 × → 右区输出）。

---

### 1. 链接线条规则

- **链接线长度**：采用"短链原则"，线条只连接「左侧卡片 → 中心圆 → 右侧卡片」的最近边缘，不得跨越大面积空白区域。
- **线条终点位置**：线条在连接目标卡片时，终点与卡片的边缘保持 **16–24px** 的间距，不可直接贴边或伸入卡片内部。
- **线条形态**：使用直线或最多 1 段折线（如果需要避让内容），避免多段折线或大角度斜线。
- **线条样式**：使用虚线或细实线（1px），颜色采用品牌低对比度辅助色（如 `#D5AE79 / 30%`透明度），不抢占视觉焦点。
- **视觉优先级**：内容 > 线条，线条只起到引导作用，不可喧宾夺主。

---

### 2. 布局分布规则

- **三大区域定义**：页面分为 **左区（输入）— 中区（核心）— 右区（输出）** 三块区域，**严格对称分布**。
- **区域宽度比例**：左区 : 中区 : 右区 = **5 : 3 : 5**（在安全区内分配），左右区域宽度一致，中区居中。
- **区域间距**：左区与中区、右区与中区之间的间距保持一致，建议 **64–96px**（本 Skill 实现使用 `gap:80px`）。
- **内容对齐**：左区卡片左对齐，右区卡片右对齐，中区元素完全居中对齐。
- **卡片垂直分布**：同一区域内的卡片等间距分布，间距一致（建议 **24–32px**），首尾卡片与区域上下边界保持一致边距。
  - 实现方式：`justify-content: space-between` + `flex:1`。
- **中心元素**：中区核心元素（如 CTR × CVR 圆形组件）垂直和水平居中，确保视觉稳定。

---

### 3. 视觉平衡原则

- **左右内容重量平衡**：左右两侧的卡片数量、尺寸、间距保持一致或接近，避免一侧拥挤、一侧稀疏。
  - 标准：左右各 3 张卡片（去掉原有的第 4 张左卡）。
- **留白一致**：三大区域的留白比例一致，不得一侧留白很大、一侧过于拥挤。
- **层级清晰**：信息层级从左到右有明确的逻辑流向，线条只是连接，不干扰阅读。

---

### 4. 禁止项

❌ 链接线跨越大面积空白区域或斜穿页面  
❌ 一侧内容过多、过密，另一侧过少、过松  
❌ 线条贴边或进入卡片内部  
❌ 使用多段折线导致线条复杂  
❌ 中区元素偏移或不居中  
❌ 左右卡片数量不对称（禁止 4:3 或 4:2 等非对称结构）

---

### 5. 实现建议

- 先划分三大区域的容器（左区/中区/右区），再放入内容。
- 使用 `display:flex; flex:5/3/5` 实现对称布局，避免绝对定位。
- 线条使用 SVG + CSS `stroke-dasharray` 实现短链连接，线条起点计算：卡片中心 y 对应 space-between 分布的 y 坐标。
- 调整内容数量或层级，保证左右区域视觉重量一致。

---

❌ 中间区域内容过少或留白过大

## R27 · 留存场景三卡对齐与平衡规则（ThreeCardLayout）

> 对应 SLIDE 8 · 场景递进路径页（次留/七留出价），参考图：「组件：留存场景（左右三块卡片）布局对齐与平衡规则」。

### 1. 整体布局规则

三块卡片必须采用「**顶部对齐、底部对齐、高度一致**」的等高布局：

```
[左卡 flex:22] [gap 32px] [中卡 flex:22] [gap 32px] [右卡 flex:56]
```

- 三卡在同一容器内水平排列，**`align-items:stretch`**，不得错位或自由拉伸。
- 卡片之间间距 `var(--space-32)`（32px，§5.7 台阶）。
- 三卡外框等高：高度差异只能通过内部弹性空间（`flex:1`）、固定底部结论区处理，不得移动整张卡片。

### 2. 卡片宽度比例规则（固定比例）

三块卡片的宽度采用固定 flex 比例：

| 卡片 | flex 比例 | 说明 |
|------|-----------|------|
| 左卡（次留出价） | `flex:22` | 重点模块，等宽于中卡 |
| 中卡（七留出价） | `flex:22` | 重点模块，等宽于左卡 |
| 右卡（详细内容） | `flex:56` | 详细内容区，较宽但等高 |

- 整体宽度保持一致，**不可随意拉宽右侧卡片**。
- 右卡虽较宽，但高度必须与左侧两张卡片**完全一致**。
- 右卡宽度不得超过 60% 或小于 50%（flex 比例锁定）。

### 3. 内容结构对齐规则

三张卡片的内容采用统一的结构和视觉节奏（自上而下）：

```
1  标题区      标题（38px bold）+ 副标题（20px muted）     ← flex-shrink:0
2  关键内容区  核心信息/列表/步骤/策略条目                  ← flex:1，space-evenly
3  KPI 结论区  大数字（100px）+ 胶囊标签                   ← flex-shrink:0（左/中卡）
```

- **标题行的基线位置必须在同一水平线**（`margin-bottom` 统一为 `var(--space-16)`）。
- 模块内的分段间距保持一致（建议 `var(--space-16)` 至 `var(--space-24)`）。
- 右侧卡片内各条内容行的高度须与左侧卡片内分段高度对应。

### 4. 视觉平衡原则

- 左侧两张卡片是「**重点模块**」，右侧卡片是「**详细内容区**」。
- 右侧卡片宽度更大，不代表高度更高；保持与左侧同等视觉重量。
- 三块卡片的色彩、圆角、边框、阴影、内边距风格一致：
  - 左卡：金色系渐变（`#FDF6EE→#F7EAD8`），border `rgba(213,174,121,0.40)`
  - 中卡：青绿系渐变（`#EEF8F6→#D6F0EC`），border `rgba(79,155,144,0.40)`
  - 右卡：纯白（`var(--white-card)`），border `rgba(11,45,58,0.08)`，轻阴影

### 5. 禁止项

| ❌ 禁止 | 原因 |
|---------|------|
| 右侧卡片高度大于或小于左侧卡片 | 破坏顶/底对齐，视觉失衡 |
| 右侧卡片顶部或底部与左侧卡片不对齐 | 参考图「错误示例」明确标注 |
| 三块卡片间距不一致 | 统一为 `var(--space-32)` |
| 右侧卡片内容留白过大或过小，导致视觉失衡 | 用 `flex:1` 均匀分配 |
| 右侧卡片宽度超过 60% 或小于 50% | flex 比例锁定在 flex:56/100 |
| 卡片标题基线不对齐 | 三卡标题 `margin-bottom` 必须统一 |

### 6. 组件树

```
ThreeCardLayout（flex-col，height:100%，padding-top:var(--space-40)）
├── 场景标签行（flex-shrink:0，28px标题 + 22px副标题 + 分隔线）
└── 三卡主区（flex:1，flex-row，gap:var(--space-32)，align-items:stretch）
      ├── 左卡 次留出价（flex:22，flex-col，金色系）

      │     ├── 中部（flex:1，space-evenly）：接入/冷启/优化阶段 + 分隔线
      │     └── 底部（flex-shrink:0）：100px大数字 + 胶囊结论标签
      ├── 中卡 七留出价（flex:22，flex-col，青绿系）
      │     ├── 标题区（flex-shrink:0）：38px标题 + 20px副标题
      │     ├── 中部（flex:1，space-evenly）：策略条目×3 + 分隔线
```

## R29 · 三指标 KPI 数据卡图表面积优化规则（ThreeMetricCardLayout）

> 对应 SLIDE 10 · 三卡成绩单页，参考图：「组件：三指标 KPI 数据卡 – 图表面积优化规则」。

### 1. 核心原则

- 图表是信息承载的核心区域，**必须突出展示**。
- 每张卡片的图表面积不得过小，保证信息平衡与视觉重心稳定。

### 2. 卡片内部垂直空间分配（固定比例）

| 区域 | 占比（建议值）| 实现方式 |
|------|-------------|---------|
| ① 顶部胶囊标签 | 约 8%  | `flex-shrink:0`，固定高度 36px |
| ② 指标标题    | 约 7%  | `flex-shrink:0`，固定 26px |
| ③ 核心数字    | 约 25% | `flex-shrink:0`，80px 大数字 |
| **④ 图表区域** | **≥ 45%** | **`flex:1; min-height:0`，禁止设 `max-height`** |
| ⑤ 结论说明    | 约 15% | `flex-shrink:0`，22px 文字 |

> **图表区域占比必须 ≥ 45%**，是本规则的核心约束。

### 3. 图表区域规则（最重要）

- 图表区域高度 **不得小于卡片内容高度的 45%**。
- 图表宽度 **不得小于卡片内容区宽度的 80%**（SVG `width:100%`）。
- 图表与上下内容的间距统一为 **16–24px**（用 `padding:var(--space-16) 0`）。
- 图表必须清晰易读，避免压缩导致信息无法识别。
- 图表类型与卡片内容匹配：趋势 → 折线图；对比 → 柱状图；占比 → 环形图。
- **禁止** 对 SVG 设置 `max-height` 限制（会导致图表面积收缩到 40% 以下）。

### 4. 卡片对齐规则

- 三张卡片**顶部对齐、底部对齐**（`align-items:stretch`）。
- 卡片高度一致，内容区域高度一致。
- 核心数字**基线对齐**（字号统一为 80px）。
- 图表区顶部起点对齐，底部落点对齐。

### 5. 间距与留白规则

- 卡片间距：`var(--space-32)`（32px，§5.7 台阶）。
- 卡片内边距：`var(--space-32) var(--space-40)`（32px 上下，40px 左右）。
- 图表周边留白适中，保证呼吸感但不浪费空间。

### 6. 禁止项（必须避免）

| ❌ 禁止 | 原因 |
|---------|------|
| 图表区域高度小于卡片高度的 40% | 图表压缩，信息失衡 |
| 图表区域宽度小于卡片内容区宽度的 70% | 图表压扁，信息难读取 |
| 对 SVG 设置 `max-height` 固定限制 | 图表被压缩在卡片底部，面积过小 |
| 图表与顶部内容距离过大（>32px），导致割离感 | `padding` 使用 §5.7 台阶值 |
| 图表不清晰，信息无法读取 | 图表 SVG 尺寸须与容器自适应 |
| 使用 `justify-content:flex-end` 把图表推底 | 图表与标题脱节，视觉失衡 |

```
      │     ├── ④ 图表区（flex:1，min-height:0，padding:16px 0）
```

```
      ├── 步骤1（flex:1，左对齐）
```

## R30 · 内容保真协议（Content Fidelity Protocol）

> **核心原则**：AI 只负责"排版美化"，不得充当内容编辑。页面视觉可以完全重做，内容必须一字不差地保留。

---

### 一、美化前：逐页建立原始内容清单

在对任何页面进行布局或视觉改造前，必须先提取该页的**原始内容清单**，包括：

| 类别 | 具体内容 |
|------|---------|
| 标题 | 页面主标题、章节标题、卡片标题 |
| 正文 | 所有段落文字、列表条目、说明文字 |
| 数字、百分比、单位 | 所有数据数值，不得改写或四舍五入 |
| 标签与注释 | Pill 标签、来源标注、注脚、角标 |
| 图片、图表及图表数据 | 坐标轴标签、图例、数据点、图表标题 |
| 流程顺序 | 步骤编号、顺序关系、连接箭头指向 |
| 信息关系 | 对比、因果、上下级、对应关系 |

---

### 二、内容保留规则

除以下三类内容可以删除外，**所有正式内容必须完整保留**：

- ✅ 明显的编辑指令（如"[待补充]"、"[TODO]"）
- ✅ 修改备注（如"[建议删除]"、"[版本说明]"）
- ✅ 占位符（如"lorem ipsum"、"此处填写..."）

---

### 三、未经明确授权，禁止的操作

未经用户明确允许，**严禁**执行以下任何操作：

| ❌ 禁止操作 | 说明 |
|------------|------|
| 删除内容 | 包括"次要信息"、"重复信息" |
| 精简内容 | 不得以"简洁美观"为由减少文字 |
| 总结替代原文 | 不得用摘要替换完整段落 |
| 改写文案 | 不得修改措辞、语序、语气 |
| 修改数字、单位、专有名词 | 数字、% 、px、人名、品牌名必须原样保留 |
| 合并不同观点 | 不得将两条观点合并为一条 |
| 隐藏次要信息 | 不得以视觉层级降低代替内容保留 |
| 用省略号代替原文 | 不得裁切长文本为"...更多" |
| 为版式美观只保留"重点" | 不得主观判断哪些内容不重要 |

---

### 四、允许调整的范围

以下操作**允许执行**，无需额外授权：

- ✅ 页面布局（分栏、分区、网格方式）
- ✅ 信息位置（内容块的空间摆放）
- ✅ 视觉层级（字号大小、颜色深浅，但不得导致内容被隐藏）
- ✅ 字体、颜色、卡片和图表样式
- ✅ 页面拆分方式（将一页拆为连续多页）

---

### 五、内容放不下时的处理顺序

当一页内容超出版面容量时，**按以下优先级处理**：

1. **优先更换合适版式**（密度更高的布局、表格、流程图等）
2. **仍放不下时拆成连续两页或多页**，保持页码延续标记
3. **不得缩小字号到难以阅读**（最小字号：正文 24px，注释 20px）
4. **不得通过删除内容解决容量问题**

---

### 六、必须保持的信息关系

以下信息关系**不得打乱或重组**：

| 信息关系类型 | 要求 |
|------------|------|
| 标题与正文的归属关系 | 标题下的正文必须归属该标题，不得错位 |
| 指标与说明的对应关系 | 数字与其单位、来源、说明必须紧密配对 |
| 流程节点的前后顺序 | 步骤 1→2→3 顺序不得颠倒或跳跃 |
| 问题与解法的对应关系 | 问题对应的解法不得错配到其他问题 |
| 图表与结论的对应关系 | 图表旁的结论必须与该图表对应 |
| 对比项、阶段和层级关系 | 左右、上下、优先级等对比结构不得打乱 |

---

### 七、美化后：逐页核对并输出检查报告

美化完成后，**必须逐页将最终页面与原始内容清单核对**，输出以下检查结果：

```
📋 内容保真检查报告
页面：[页面标题 / 编号]
────────────────────────────────
原始正式内容数量：   X 条
已保留内容数量：     X 条
遗漏内容：           [列出]
被改写内容：         [列出原文 → 改后文]
新增但无来源的内容：  [列出]
被拆分到其他页面的内容：[列出，含目标页]
```

---

### 八、完成标准（硬约束）

只有满足以下全部条件，才允许标记为"完成"：

| 检查项 | 通过标准 |
|--------|---------|
| 正式内容遗漏 | **0 条** |
| 未授权改写 | **0 处** |
| 数字和单位错误 | **0 处** |
| 信息关系改变 | **0 处** |
| 无来源新增内容 | **0 条** |

**任何一项不为 0，必须返回修改，不得直接交付。**

## R31 · MetricCardGroup 指标卡片组宽度规则（硬约束）

> **适用范围**：页面包含 2～4 个同级 KPI / 指标卡片时，本节规则**优先于** R16 §二"禁止无限拉宽文本卡片"，直接生效。

---

### 一、页面标题居中（相对画布）

页面标题必须相对于**完整画布**水平居中，不得相对于 Logo 位置、左侧内容列或卡片组居中。

```css
/* ✅ 正确 */
.page-title-row { left: 0; right: 0; text-align: center; }

/* ❌ 禁止 */
.page-title-row { text-align: left; margin-left: 220px; }
```

---

### 二、MetricCardGroup 占满安全区可用宽度

```css
.metric-card-group {
  position: absolute;
  left: 220px;
  right: 220px;
  width: auto;
  margin: 0;
  box-sizing: border-box;
}
```

---

### 三、同级指标卡片使用共同 Grid 父容器

```css
.metric-card-grid {
  display: grid;
  grid-template-columns: repeat(N, minmax(0, 1fr)); /* N = 卡片实际数量 */
  gap: 24px;   /* 允许 24px～40px */
  align-items: stretch;
}
```

- `N` 由内容决定，不得写死；
- 所有列必须等宽（`1fr`），不得单独扩大某列；
- `align-items: stretch` 使所有卡片等高，不得改为 `flex-start`。

---

### 四、禁止项

| ❌ 禁止操作 | 说明 |
|------------|------|
| 为同级指标卡设置固定 px 宽度 | 导致卡片组整体收缩 |
| `justify-content: flex-start` | 导致卡片靠左堆积 |
| 将卡片组锚定在页面左侧 | 违反左右对称要求 |
| 用缩窄外框解决行长问题 | 行长约束只作用于内部文字区 |

---

### 五、文字行长约束只作用于卡片内部

```css
/* ✅ 正确：行长约束只作用于内部文字区 */
.metric-card-copy { max-width: 32ch; margin-inline: auto; }

/* ❌ 禁止：通过缩窄卡片外框限制行长 */
.metric-card { max-width: 600px; }
```

---

### 六、左右对称检查（硬约束）

- 卡片组左右边缘必须基本对称；
- 单侧连续无功能空白 **> 安全区宽度的 20%**（> 651px）→ 布局失败，必须重新生成；
- 验证：`卡片组 offsetLeft − 220` ≈ `3476 − (卡片组 offsetLeft + 卡片组 offsetWidth)`，误差 ≤ 20px。

---

### 七、KPI 卡片横向自适应例外声明

KPI / 指标卡片组允许横向自适应展开至安全区全宽；R16 §二"禁止无限拉宽文本卡片"**不适用于**内部内容居中的同级 MetricCard：

- 每张卡片内部已有居中文字区（`margin-inline: auto`）；
- 宽度由 Grid `1fr` 等比分配，不存在单卡无限拉宽；
- 横向空间由所有卡片共同吸收，视觉均衡。

---

### 八、检查清单

```text
[ ] 页面标题是否相对完整画布居中，而非相对 Logo 或卡片组居中
[ ] MetricCardGroup 是否 left:220px; right:220px（占满安全区宽度）
[ ] 卡片是否使用 grid-template-columns: repeat(N, minmax(0, 1fr))
[ ] 是否未为同级指标卡设置固定 px 宽度
[ ] 是否未使用 justify-content: flex-start
[ ] 是否未将卡片组锚定在页面左侧
[ ] 文字行长 max-width 是否只作用于 .metric-card-copy，而非卡片外框
[ ] 单侧空白是否未超过安全区宽度的 20%（> 651px 判定失败）
```

## R32 · 来源图片与插图保留协议（P0 硬约束）

> **优先级**：本协议优先级高于页面低信息密度规则、视觉风格统一规则、模板槽位容量规则、页面美化规则、内容精简规则和图片占位符规则。

本协议适用于用户上传的 PPTX、PDF、Word、图片型页面及其他包含视觉素材的正式来源文件。

除非用户明确要求删除或替换来源图片，否则必须**默认保留**所有具有内容、证据、说明或品牌识别价值的来源视觉素材。

---

### 一、来源图片默认保留原则

来源文件中出现的图片**不得默认视为装饰**。

以下类型必须默认标记为 `must_preserve`：

| 类别 | 示例 |
|------|------|
| 产品 / 功能截图 | 界面截图、数据看板截图、产品功能截图 |
| 人物图片 | 人物照片、人物插图、出席嘉宾图 |
| 品牌 / 案例视觉 | 品牌案例图片、活动现场图片、营销 / 广告素材 |
| 信息图 | 流程示意图、架构示意图、业务关系图 |
| 数据可视化图片 | 图表图片（来源截图，非重新绘制） |
| 品牌标识 | 品牌 Logo（独立出现）、奖项、证书、报告封面 |
| 二维码 | 任何二维码 |
| 证据图片 | 用于证明正文结论的视觉证据 |
| 主图 | 原始页面中占据明显视觉面积的主图 |
| 语义关联图标 | 与正文存在对应关系的图标或插图 |
| 语义不完整判定 | 删除后会导致页面语义、证据或叙事不完整的图片 |

以下素材可以标记为 `decorative`（允许省略或替换）：

- 无信息含义的背景纹理
- 纯装饰色块 / 无业务语义的光效 / 普通分隔线
- 可由新风格背景替代的抽象装饰
- 与正文无对应关系的重复小装饰

**只有标记为 `decorative` 的素材，才允许在风格重构时省略或替换。**

---

### 二、无法判断时的默认行为

当无法判断图片是否具有内容价值时，**必须默认标记为 `must_preserve`**。

不得以"看起来像装饰"、"模板图"、"风格不统一"为由单方面标记为 `decorative`。

---

### 三、违反本协议的判定标准（任一即为失败）

| 违规行为 | 说明 |
|---------|------|
| 静默删除来源图片 | 来源中的 `must_preserve` 图片未出现在输出 HTML 中 |
| 用占位符替代来源图片 | 用灰色方块、`[图片待补充]` 或通用图标替代真实来源图片 |
| 用 AI 生成图替代来源图片 | 未经用户许可，用 AIGC 图替换真实截图、产品图或人物图 |
| 图片缩小到不可辨认 | 来源图片被压缩到 80px 以下或肉眼无法辨认 |
| 将图片移入背景层 | 将具有信息价值的图片降级为背景图、水印或装饰层 |

---

### 四、检查清单

```text
[ ] 所有来源中的 must_preserve 图片是否出现在输出 HTML 中
[ ] 是否未用占位符、灰色方块或 AIGC 图替代来源图片
[ ] 来源图片最终渲染尺寸是否可辨认（最小边 ≥ 80px）
[ ] 具有信息价值的图片是否作为独立 DOM 节点渲染，未移入背景层
[ ] 不确定价值的图片是否已默认标记为 must_preserve
```

## R33 · 页面级纵向容量预算与溢出检测协议（P0 硬约束）

> **唯一来源**：`orchestrator/rules/layout_capacity_protocol.md`。本条为摘要速查。
> **优先级**：高于卡片等高规则、卡片自动高度规则、模板默认布局、低信息密度规则。

---

### 一、两类溢出必须分别检查

| 类型 | 定义 |
|------|------|
| 卡片内部溢出 | 内容超出单张卡片内部边界 |
| **页面级溢出** | 卡片/模块超出安全区，或被画布 overflow:hidden 裁切 |

不得因为卡片内部没有溢出就判定整个页面没有溢出。

---

### 二、页面可用高度必须显式计算

```
bodyAvailableHeight
= 828 - pageHeaderHeight - headerToBodyGap
  - optionalFooterHeight - bodyPadding(top+bottom)
```

page_plan 必须记录 `body_available_height`。未记录则判定 `layout-capacity-not-calculated`。

---

### 三、纵向卡片组容量预估

```
stackRequiredHeight
= Σ卡片预估高度 + (N-1) × gap + 上下内边距

必须满足：stackRequiredHeight <= bodyAvailableHeight - 16px
```

不满足时必须切换布局或拆页，不得继续用当前纵向结构。

---

### 四、左右分栏共享同一主体内容带

```
top(left) = top(right)  ·  bottom(left) = bottom(right)
```

页面标题必须放在统一标题区，不得只占左侧栏纵向空间。

---

### 五、四个同级模块优先 2 × 2 矩阵

四行纵向堆叠只有在 `stackRequiredHeight ≤ bodyAvailableHeight × 0.9` 时才允许；否则：

```
优先：2 × 2 矩阵 > 两列双行 > 拆页
```

---

### 六、正式内容禁止静默裁切

`overflow:hidden`、`line-clamp`、`display:none`、负 margin、transform 上移等**不得用于解决溢出**。

`overflow:hidden` 仅允许用于：背景装饰 / 图片等比裁切 / 图表绘图区 / 纯装饰元素。

---

### 七、最后一项完整可见检查

```
lastItemRect.bottom <= bodyBandRect.bottom - 2px
```

最后一张卡片必须完整可见；仅部分可见则判定为页面溢出。

---

### 八、失败代码速查

```text
page-content-outside-safe-zone  · page-bottom-overflow
vertical-stack-height-exceeded   · formal-content-clipped
column-height-mismatch           · body-band-height-undefined
layout-capacity-not-calculated   · last-card-partially-visible
scroll-height-exceeds-client-height / scroll-width-exceeds-client-width
formal-content-moved-outside-canvas
layout-fixed-by-negative-margin  · layout-fixed-by-transform
```

---

### 九、检查清单（生成前）

```text
[ ] 是否计算 body_available_height 并写入 page_plan
[ ] 纵向卡片组 stackRequiredHeight ≤ bodyAvailableHeight - 16px
[ ] 左右两列是否共享同一主体内容带
[ ] 四个同级模块是否优先评估 2 × 2 矩阵
[ ] 未通过容量检查时是否已切换布局或拆页
[ ] 未使用 overflow:hidden / 负margin / transform 修补溢出
```

### 十、检查清单（生成后）

```text
[ ] 所有正式内容节点 bottom ≤ safeZoneRect.bottom
[ ] 左右两列顶部 / 底部对齐（误差 ≤ 2px）
[ ] 最后一张卡片完整可见（bottom ≤ bodyBandRect.bottom - 2px）
[ ] 所有正式节点 scrollHeight ≤ clientHeight
[ ] 未使用 line-clamp / overflow:hidden 隐藏正式内容
[ ] 100% 画布与缩放预览均无溢出
```
